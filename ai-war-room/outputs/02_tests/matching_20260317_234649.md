# Plan de Pruebas: Superficie `matching`

Este documento detalla el plan de pruebas para la superficie `matching`. Su objetivo es identificar brechas en la cobertura, proponer pruebas de alto valor y garantizar que la funcionalidad crítica sea robusta, segura y esté alineada con los documentos de producto.

## 1. Evaluación de Cobertura

Tras analizar el código y la documentación, la cobertura actual es críticamente baja. No existen pruebas unitarias ni de integración para el endpoint de `matching`, que es una funcionalidad central del producto.

-   **Pruebas Unitarias:** **Ausentes.** Las funciones auxiliares de puntuación (`haversineKm`, `recencyScore`, `distanceScore`, `colorOverlap`) son puras y fácilmente testeables, pero no tienen ninguna prueba. Esto representa un riesgo significativo ya que la lógica de negocio principal no está verificada.
-   **Pruebas de Integración:** **Ausentes.** No hay pruebas que verifiquen el flujo completo del endpoint `GET /api/matching/cases/:id`, incluyendo la interacción con la base de datos, la lógica de autenticación, el manejo de errores y la estructura de la respuesta.
-   **Comprobaciones de Auth y Permisos:** El endpoint utiliza el middleware `verifyToken`, pero no hay pruebas que confirmen que las solicitudes no autenticadas sean rechazadas.
-   **Cobertura de Modos de Fallo:** El código incluye un bloque `try/catch`, pero no se prueba ningún escenario de fallo, como un ID de caso no encontrado (`404`), un fallo en la consulta a la base de datos (`500`) o datos de entrada malformados.
-   **Desajustes Doc/Código:** Se han identificado desajustes críticos:
    1.  **Lógica de IA (Metadatos vs. Imágenes):** El documento `AI_MATCHING.md` describe un sistema centrado en la "similitud de imágenes". La implementación actual (`server/routes/matching.ts`) es **100% basada en metadatos** (ubicación, fecha, color, tamaño, rasgos). No realiza ningún procesamiento de imágenes. Esta es una divergencia fundamental que debe ser visibilizada.
    2.  **Duplicación de Lógica Frontend/Backend:** El archivo `src/app/utils/matching.ts` (frontend) reimplementa la lógica de puntuación del backend. Esta duplicación, con sutiles diferencias, es una fuente segura de inconsistencias y errores.
    3.  **Contrato de API:** El backend devuelve un objeto con `caseId`, `confidence`, `location`, etc. El cliente de frontend (`matchingReal.ts`, `api.ts`) parece esperar una estructura anidada `{ case: Case, ... }`. Esto provocará un fallo en la integración del cliente.

## 2. Pruebas Faltantes de Mayor Valor

Dada la falta total de cobertura, se debe priorizar la creación de pruebas que cubran los flujos críticos y la lógica de negocio principal.

-   **(Bloqueante)** **Prueba de Integración - Flujo Exitoso:** Validar el camino feliz completo del endpoint `GET /api/matching/cases/:id`, desde la solicitud hasta una respuesta 200 con resultados clasificados.
-   **(Bloqueante)** **Pruebas Unitarias - Funciones de Puntuación:** Crear pruebas para `recencyScore`, `distanceScore` y `colorOverlap` para garantizar que la lógica de clasificación central sea correcta.
-   **(Bloqueante)** **Prueba de Integración - Autenticación:** Asegurar que el endpoint esté debidamente protegido y rechace las solicitudes sin un token válido.
-   **(Alto Valor)** **Prueba de Integración - Caso No Encontrado:** Verificar el manejo correcto del error 404 cuando el ID del caso de origen no existe.
-   **(Alto Valor)** **Prueba de Integración - Lógica de Filtrado:** Confirmar que los candidatos se filtran correctamente por tipo, estado (`active`), ID (para no coincidir consigo mismo) y umbral de confianza.
-   **(Desajuste de Documentación)** **Prueba de Regresión - Comportamiento Actual:** Crear una prueba que confirme explícitamente que la puntuación se basa únicamente en metadatos, para que sirva como registro del comportamiento actual y resalte la brecha con la documentación.

## 4. Estrategia de Mocking

-   **Dependencia Principal:** La base de datos, a través de la función `query` exportada desde `../db.js`.
-   **Herramienta:** Se recomienda `jest` (`jest.mock`, `jest.spyOn`).
-   **Implementación:**
    1.  Utilizar `jest.mock('../db.js', ...)` para reemplazar el módulo de la base de datos.
    2.  Antes de cada prueba, configurar el comportamiento del mock de `query`. Se necesitarán al menos dos configuraciones por prueba: una para la consulta del caso de origen (`SELECT ... WHERE id = $1`) y otra para la consulta de candidatos (`SELECT ... WHERE type = ANY($1::text[])`).
    3.  Ejemplo: `query.mockResolvedValueOnce({ rows: [originCaseFixture] }).mockResolvedValueOnce({ rows: [candidate1, candidate2] });`
    4.  Para las pruebas que dependen del tiempo (`recencyScore`), se debe mockear `Date.now()` para asegurar resultados deterministas. Ejemplo: `jest.spyOn(Date, 'now').mockReturnValue(new Date('2026-03-20T10:00:00.000Z').getTime());`

## 5. Fixtures de Datos Necesarios

Se deben crear objetos JS que representen registros de la tabla `cases`. Estos se usarán para alimentar el mock de la base de datos.

-   `originCaseLost`: Un caso de tipo 'lost' con ID=1, ubicación, colores, tamaño y rasgos definidos.
-   `originCaseFound`: Un caso de tipo 'found' para probar el flujo inverso.
-   `perfectMatchCandidate`: Un candidato de tipo 'found' muy reciente, cercano geográficamente y con todos los metadatos coincidentes. Debería generar una `confidence` > 90.
-   `partialMatchCandidate`: Un candidato que solo coincide en color y tamaño, pero es antiguo y lejano.
-   `lowConfidenceCandidate`: Un candidato sin coincidencias, antiguo y lejano. Debería puntuar por debajo de 15.
-   `inactiveCandidate`: Un candidato con `status: 'resolved'`.
-   `sameTypeCandidate`: Un candidato de tipo 'lost' para probar que es filtrado cuando el origen es 'lost'.
-   `caseWithNullGeo`: Un candidato con `approx_lat: null` y `approx_lng: null`.
