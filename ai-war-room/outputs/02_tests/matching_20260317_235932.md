# Plan de Pruebas: Superficie de Matching de PETTODO

## 1. Evaluación de Cobertura

Una revisión de los ficheros proporcionados revela brechas de cobertura significativas que deben ser abordadas para garantizar la fiabilidad y corrección de la funcionalidad de matching.

-   **Pruebas Unitarias Faltantes:** No hay pruebas unitarias para ninguna de las funciones de ayuda de puntuación en `server/routes/matching.ts` (ej. `recencyScore`, `distanceScore`, `colorOverlap`). Estas funciones de lógica pura son candidatas ideales para pruebas unitarias pero carecen de ellas por completo. Lo mismo ocurre con las funciones duplicadas en el frontend (`src/app/utils/matching.ts`).

-   **Pruebas de Integración Faltantes:** El endpoint principal de la API, `GET /api/matching/cases/:id`, no tiene ninguna prueba de integración. La interacción entre la lógica de la ruta, el middleware `verifyToken` y las consultas a la base de datos (`db.query`) es completamente ciega. Esta es la brecha más crítica.

-   **Cobertura de Modos de Fallo:** El manejo de errores existe en el código (`try/catch`, `404` si no se encuentra el caso), pero no está verificado por ninguna prueba. Los fallos de la base de datos, la entrada de datos nulos o malformados, y los errores de autenticación no están cubiertos.

-   **Comprobaciones de Autenticación y Permisos:** El endpoint utiliza el middleware `verifyToken`, pero no hay pruebas que confirmen que las solicitudes no autenticadas son rechazadas correctamente con un `401 Unauthorized`.

-   **Discrepancias entre Documentación y Código:**
    1.  **Lógica Duplicada:** Existe una divergencia peligrosa entre la lógica de puntuación del backend (`server/routes/matching.ts`) y una implementación similar pero sutilmente diferente en el frontend (`src/app/utils/matching.ts`). Por ejemplo, los umbrales de confianza (15 en backend, 20 en frontend) y los cálculos de puntuación para colores y rasgos difieren.
    2.  **Visión vs. Realidad:** El documento `AI_MATCHING.md` describe un pipeline de IA de varias etapas (puerta de calidad, extracción de embeddings) que no se corresponde con la implementación heurística actual basada en reglas.
    3.  **Etiquetas de Coincidencia:** La documentación especifica bandas de coincidencia para el usuario ("Probable Match", "Strong Match"), mientras que la API devuelve una puntuación de `confidence` numérica. Falta la lógica para mapear esta puntuación a las etiquetas aprobadas.

## 2. Pruebas Faltantes de Mayor Valor

Para maximizar el impacto y reducir el riesgo, debemos priorizar las siguientes pruebas.

1.  **[Bloqueante] Prueba de Integración para `GET /api/matching/cases/:id`:** Una prueba completa que simule una llamada a la API, mockee la base de datos y verifique que un conjunto de casos candidatos se clasifica y devuelve correctamente. Esto valida el flujo de trabajo crítico de la superficie.

2.  **[Bloqueante] Pruebas Unitarias para los Helpers de Puntuación (Backend):** Pruebas para `recencyScore`, `distanceScore`, `colorOverlap`, etc. Esto asegura que el núcleo del algoritmo de clasificación es correcto y estable, protegiéndolo contra regresiones.

3.  **[Bloqueante] Pruebas de Modos de Fallo y Seguridad para la API:** Casos de prueba que confirmen el comportamiento correcto cuando el caso de origen no existe (`404`), cuando el token de autenticación falta o es inválido (`401`), y cuando la base de datos falla (`500`).

4.  **[Bloqueante] Prueba de Contrato de Respuesta de la API:** Una prueba para asegurar que la estructura de la respuesta de la API (`caseId`, `confidence`, `reasons`, `caution`, `nextAction`) se adhiere a los requisitos del producto definidos en la documentación.

5.  **[Recomendable] Prueba de Consistencia de Lógica:** Identificar la divergencia entre la lógica de puntuación del frontend y del backend. La prueba debe afirmar que la lógica del backend es la única fuente de verdad, señalando la redundancia del frontend como un problema técnico a resolver.

## 4. Estrategia de Mocking

Para ejecutar estas pruebas de forma aislada y determinista, se requiere una estrategia de mocking clara.

-   **`db.query`**: Debe ser mockeado usando un framework como `jest.mock`. Para cada prueba de integración, el mock debe ser configurado para devolver respuestas específicas:
    -   Una llamada para el caso de origen (`SELECT * FROM cases WHERE id = $1`).
    -   Una segunda llamada para los candidatos (`SELECT * FROM cases WHERE type = ANY(...)`).
    -   Se puede configurar para que devuelva un array vacío o lance un error para probar los modos de fallo.

-   **`verifyToken` (Middleware):** Este middleware debe ser mockeado. Para las pruebas de éxito, el mock simplemente llamará a `next()` y opcionalmente añadirá un usuario simulado al objeto `req`. Para las pruebas de fallo de autenticación, el mock devolverá directamente una respuesta `401`.

-   **`Date.now()`**: Crítico para probar `recencyScore`. Debe ser mockeado a un valor constante en las pruebas unitarias y de integración para que los cálculos de tiempo sean predecibles. `jest.spyOn(Date, 'now').mockReturnValue(...)` es ideal para esto.

## 6. Orden de Ejecución

Se recomienda el siguiente orden para implementar el plan de pruebas y abordar los riesgos de manera incremental.

1.  **Implementar Pruebas Unitarias (Bloqueante):** Comenzar con las pruebas para las funciones puras de puntuación en `server/routes/matching.ts`. Son rápidas de escribir, no tienen dependencias externas y validan el corazón del algoritmo.

2.  **Implementar Pruebas de Fallo y Seguridad de la API (Bloqueante):** Escribir las pruebas para los casos `401`, `404` y `500`. Esto asegura que las protecciones básicas y el manejo de errores estén funcionando antes de probar la lógica de negocio.

3.  **Implementar Pruebas de Integración "Happy Path" (Bloqueante):** Abordar los casos de éxito principales, como la coincidencia perfecta y parcial. Esto verifica que la funcionalidad principal cumple con los requisitos.

4.  **Implementar Pruebas de Conformidad con la Documentación (Bloqueante):** Añadir las aserciones que garantizan que el contrato de la API (campos obligatorios, mensaje de precaución) se respeta.

5.  **Crear Tarea Técnica para Sincronizar Lógica FE/BE:** La divergencia de lógica entre el frontend y el backend es un riesgo de mantenimiento crítico. Se debe crear un ticket para **eliminar la función `rankMatches` del frontend** y confiar únicamente en la API del backend como la fuente de verdad para el matching. No es una prueba, sino una acción correctiva clave derivada de este análisis.

6.  **Implementar Pruebas de Robustez (Recomendable):** Una vez que la funcionalidad principal esté cubierta, añadir pruebas para casos extremos como datos nulos o malformados para fortalecer el sistema.
