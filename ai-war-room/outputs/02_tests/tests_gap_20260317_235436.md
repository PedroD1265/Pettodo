# Plan de Pruebas de PETTODO para `tests_gap`

Este documento detalla un plan de pruebas práctico para cerrar las brechas de cobertura críticas identificadas en el backend, con un enfoque específico en la funcionalidad de `matching` y las rutas de `images` que carecen de pruebas.

## 1. Evaluación de Cobertura

La base de pruebas actual (`auth`, `cases`, `pets`, `public`, `import`) proporciona una cobertura de integración básica para las operaciones CRUD en entidades principales. Sin embargo, existen brechas significativas:

-   **Ausencia Total de Pruebas para Matching**: El archivo `server/routes/matching.ts`, que implementa una lógica de negocio crítica y compleja para la búsqueda de coincidencias, no tiene **ningún archivo de prueba asociado (`tests/matching.test.ts`)**. Esta es la brecha más crítica.
-   **Cobertura Incompleta para Imágenes**: Las pruebas para `server/routes/images.ts` existen (`tests/images.test.ts`), pero omiten funcionalidades clave:
    -   No hay pruebas para la lógica de asociación de imágenes con `cases` (solo se prueba para `pets`) en la generación de URLs de subida.
    -   La ruta `DELETE /api/cases/:id/images/:imageId` no está probada en absoluto.
-   **Discrepancia entre Documentación y Código**: El documento `AI_MATCHING.md` especifica que los resultados de matching deben presentarse como bandas de confianza (`Probable Match`, `Strong Match`), mientras que la API devuelve un puntaje numérico de `confidence`. Esto debe ser validado y señalado.
-   **Rutas Completamente Ignoradas**: Rutas como `protected-contact`, `community-dogs`, `change-requests`, `evidence`, `reviews`, y `abuse` están montadas en `app.ts` pero carecen de cualquier tipo de prueba. Aunque fuera del enfoque inmediato de este bundle, su ausencia representa un riesgo.

El enfoque principal de este plan será crear la cobertura para `matching.ts` desde cero y completar las pruebas faltantes para `images.ts`.

## 2. Pruebas Faltantes de Mayor Valor

Se priorizan las siguientes pruebas para maximizar el retorno de la inversión en calidad y mitigar los mayores riesgos.

1.  **Suite de Pruebas de Integración para `GET /api/matching/cases/:id` (BLOQUEANTE)**: Esta es una funcionalidad central sin red de seguridad. Probar su lógica de puntuación, manejo de errores y autorización es fundamental antes de cualquier despliegue.
2.  **Pruebas para `DELETE /api/cases/:id/images/:imageId` (BLOQUEANTE)**: La ausencia de pruebas para una operación de borrado es un riesgo alto. Es crucial asegurar que solo el propietario pueda borrar y que tanto la referencia en la base de datos como el blob en el almacenamiento se eliminen.
3.  **Pruebas de Autorización para Carga de Imágenes de Casos (Alta Prioridad)**: Validar que un usuario solo puede generar URLs de subida para casos que le pertenecen (`entityType: 'case'`).
4.  **Pruebas de Modos de Fallo para Matching (Deseable)**: Casos como datos de entrada nulos (coordenadas, colores), fallos de base de datos y escenarios sin candidatos válidos.
5.  **Prueba de Discrepancia de Documentación (Informativa)**: Una prueba que verifique el formato de respuesta de matching y sirva como punto de partida para alinear el código con la especificación del producto.

## 4. Estrategia de Mocking

La estrategia seguirá el patrón existente en el repositorio, utilizando `vitest` y `supertest`.

-   **Base de Datos (`db.js`)**: Se utilizará `vi.hoisted` y `vi.mock` para reemplazar la función `query` con un mock de `vi.fn()`. Cada test controlará los datos de retorno (`mockQuery.mockResolvedValue({ rows: [...] })`) para simular diferentes escenarios de la base de datos. Para `matching.ts`, es clave usar `mockResolvedValueOnce` para simular la secuencia de queries (obtener origen, luego obtener candidatos).
-   **Autenticación (`firebaseAdmin.js`)**: Se mockeará `adminAuth.verifyIdToken` para devolver un usuario falso (`FAKE_USER`) en las rutas protegidas o para simular un fallo de autenticación (`mockRejectedValue`).
-   **Almacenamiento de Blobs (`blobStorage.js`)**: Se mockearán `isStorageConfigured`, `generateUploadSasUrl`, `generateReadSasUrl` y `deleteBlob` para aislar las pruebas de la infraestructura real de Azure y verificar que son llamadas con los argumentos correctos.

## 5. Fixtures de Datos Necesarios

Se definirán fixtures reutilizables al inicio de los archivos de prueba para mantener la legibilidad y consistencia.

-   **`FAKE_USER` / `FAKE_USER_2`**: Objetos que simulan un usuario autenticado vía Firebase.
-   **`MOCK_ORIGIN_CASE_LOST`**: Un objeto de caso de tipo `lost` con todos los campos necesarios para el cálculo de puntuación (coordenadas, tamaño, colores, etc.).
-   **`MOCK_ORIGIN_CASE_FOUND`**: Similar al anterior, pero de tipo `found`.
-   **`MOCK_CANDIDATE_CASES_ARRAY`**: Un array de objetos de caso que servirán como candidatos. Debe incluir una variedad de datos para probar la lógica de puntuación: una coincidencia fuerte, una débil y una con datos faltantes.
-   **`MOCK_CASE_IMAGE_ROW`**: Un objeto que simula una fila de la tabla `case_images`, necesario para probar la lógica de borrado.

## 6. Orden de Ejecución

Se recomienda abordar la implementación de las pruebas en el siguiente orden de prioridad:

1.  **Pruebas Bloqueantes para `matching.ts`**: Implementar los tests básicos de autorización (401), caso no encontrado (404), el camino feliz (200 con resultados) y el manejo de errores de base de datos (500).
2.  **Pruebas Bloqueantes para `images.ts`**: Implementar la suite completa para `DELETE /api/cases/:id/images/:imageId` y las pruebas de validación de pertenencia para `POST /storage/upload-url` con `entityType: 'case'`.
3.  **Pruebas de Casos Borde para `matching.ts`**: Añadir pruebas para escenarios sin candidatos, candidatos por debajo del umbral de confianza y manejo de datos nulos.
4.  **Prueba de Discrepancia con Documentación**: Implementar el test que valida el formato de `confidence` para registrar formalmente la desviación del diseño. Este test puede ser marcado como "esperado que falle" o simplemente documentar el comportamiento actual.
