# Plan de Pruebas para PETTODO: `tests_gap`

Este documento detalla un plan de pruebas práctico para cerrar las brechas de cobertura en el repositorio, con un enfoque específico en las rutas y la lógica de negocio críticas que actualmente carecen de pruebas automatizadas.

## 1. Evaluación de Cobertura

La base de pruebas actual (`vitest` + `supertest`) es sólida para las operaciones CRUD de entidades como `pets`, `cases` e `images`, y también cubre la autenticación básica. Sin embargo, se ha identificado una brecha crítica y de alto riesgo:

- **Cobertura Nula en `server/routes/matching.ts`**: El archivo que implementa la lógica de emparejamiento de casos (`GET /api/matching/cases/:id`) tiene un **0% de cobertura de pruebas**. Esta es la funcionalidad más compleja y distintiva del producto, y su falta de pruebas representa un riesgo significativo de regresión y comportamiento inesperado.
- **Funciones Puras sin Probar**: Las funciones auxiliares de puntuación heurística (`haversineKm`, `recencyScore`, `distanceScore`, `colorOverlap`) dentro de `matching.ts` son candidatas perfectas para pruebas unitarias, pero actualmente no se están probando.
- **Cobertura de Permisos**: Aunque las pruebas existentes validan la propiedad (p. ej., un usuario no puede modificar la mascota de otro), esta validación falta por completo en el módulo de `matching`. La ruta actual permite a *cualquier* usuario autenticado ejecutar un emparejamiento en *cualquier* caso, lo cual coincide con el comentario en el código, pero debería ser validado explícitamente por una prueba.

## 2. Pruebas Faltantes de Mayor Valor

A continuación se priorizan las pruebas más valiosas que deben implementarse para mitigar los riesgos identificados.

1.  **Pruebas Unitarias para los Helpers de Puntuación (Bloqueante)**: Probar las funciones puras en `matching.ts` es un paso fundamental y de bajo esfuerzo. Asegura que los componentes básicos del algoritmo de emparejamiento funcionen como se espera de forma aislada.
2.  **Suite de Integración para `GET /api/matching/cases/:id` (Bloqueante)**: Esta es la máxima prioridad. Se necesita una suite completa que cubra la autenticación, los casos de éxito, los modos de fallo y la lógica de negocio central.
3.  **Pruebas de Regresión para la Lógica de Clasificación (Crítico)**: Pruebas específicas que aseguren que los cambios en los pesos de la puntuación o en la lógica no degraden la calidad de los resultados. Por ejemplo, un candidato "muy cercano" siempre debe tener una puntuación de distancia mayor que uno "lejano".
4.  **Pruebas de Contrato para la Estructura de Respuesta (Recomendado)**: Validar que la estructura de la respuesta de la API de emparejamiento (`confidence`, `reasons`, `nextAction`, etc.) se mantenga consistente.
5.  **Prueba de Discrepancia Documentación/Código (Recomendado)**: El documento `AI_MATCHING.md` menciona bandas de coincidencia como "Probable Match", mientras que el código devuelve una `confidence` numérica. Una prueba debe afirmar el comportamiento actual para que cualquier cambio futuro sea intencional.

```typescript
// tests/matching.test.ts

describe('Matching Logic Helpers', () => {
  // test: recencyScore debería dar más puntos a los casos más recientes
  it('recencyScore should score recent cases higher', () => {
    // Arrange
    const twoHoursAgo = Date.now() - (1000 * 60 * 60 * 2);
    const threeDaysAgo = Date.now() - (1000 * 60 * 60 * 72);
    // Act
    const scoreRecent = recencyScore(twoHoursAgo);
    const scoreOld = recencyScore(threeDaysAgo);
    // Assert
    expect(scoreRecent.score).toBe(22); // o 30 si es menos de 2h
    expect(scoreOld.score).toBe(2);
    expect(scoreRecent.score).toBeGreaterThan(scoreOld.score);
  });

  // test: distanceScore debería dar más puntos a las distancias más cortas
  it('distanceScore should score closer distances higher', () => {
    // Arrange & Act
    const scoreClose = distanceScore(0.4); // < 0.5km
    const scoreFar = distanceScore(4.5);   // < 5km
    // Assert
    expect(scoreClose.score).toBe(30);
    expect(scoreFar.score).toBe(8);
    expect(scoreClose.score).toBeGreaterThan(scoreFar.score);
  });

  // test: colorOverlap debería contar coincidencias sin importar mayúsculas/minúsculas
  it('colorOverlap should count matches case-insensitively', () => {
    // Arrange
    const originColors = ['Blanco', 'NEGRO'];
    const candidateColors = ['Negro', 'marrón'];
    // Act
    const overlap = colorOverlap(originColors, candidateColors);
    // Assert
    expect(overlap).toBe(1);
  });
});
```

## 4. Estrategia de Mocking

La clave para probar `matching.ts` es simular la base de datos para controlar los datos de entrada del algoritmo.

- **`vi.mock('../server/db.js')`**: Se debe hacer mock del módulo `db.js`, específicamente de la función `query`.
- **`mockQuery.mockResolvedValueOnce(...)`**: Cada llamada a `supertest` que involucre la base de datos requerirá una configuración del mock de `query`.
    1.  La **primera llamada** a `mockQuery` en un test de `GET /api/matching/cases/:id` debe simular la respuesta de `SELECT * FROM cases WHERE id = $1`. Se le debe pasar un array con el "caso de origen".
    2.  La **segunda llamada** debe simular la respuesta de `SELECT * FROM cases WHERE type = ANY($1::text[]) ...`. Se le debe pasar un array con los "casos candidatos".
- **`vi.mock('../server/firebaseAdmin.js')`**: Como en los otros archivos de prueba, se debe hacer mock de `verifyIdToken` para simular un usuario autenticado y evitar llamadas reales a Firebase.
