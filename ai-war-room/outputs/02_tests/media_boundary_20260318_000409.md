# Plan de Pruebas: `media_boundary`

Este documento detalla el plan de pruebas para la superficie `media_boundary`, que abarca la generación de URLs de almacenamiento, la persistencia de referencias de imágenes y la cobertura de pruebas de la API relacionada.

## 1. Evaluación de Cobertura

La cobertura de pruebas actual, encontrada en `tests/images.test.ts`, es un buen punto de partida pero presenta lagunas significativas:

-   **Rutas de `pet`:** Tienen una cobertura básica para los "caminos felices" (éxito), validación de propiedad y algunos errores de validación.
-   **Rutas de `case`:** La cobertura es muy deficiente. Las pruebas para `POST` y `GET` son mínimas, y **no existen pruebas para `DELETE /api/cases/:id/images/:imageId`**. Esta disparidad es un riesgo.
-   **Pruebas Unitarias:** Funciones auxiliares cruciales como `safeName` y `blobExt` en `server/routes/images.ts` carecen por completo de pruebas unitarias, lo que las hace vulnerables a regresiones.
-   **Manejo de Fallos:** No hay pruebas que verifiquen el comportamiento de la API cuando las dependencias externas (base de datos, Azure Blob Storage) fallan. Los bloques `try/catch` existen, pero su comportamiento no está validado.
-   **Comprobaciones de Permisos Avanzadas:** La validación de propiedad es buena, pero no se cubren casos de borde, como intentar asociar una imagen de un tipo de entidad (`case`) a otra (`pet`).

## 2. Pruebas Faltantes de Mayor Valor

Priorizando el impacto y el riesgo, las siguientes pruebas son las más importantes a añadir:

1.  **[BLOQUEANTE] Cobertura completa para la API de Imágenes de `case`:** Es fundamental alcanzar la paridad con la API de `pet`, especialmente para el endpoint `DELETE` que no tiene pruebas, y añadir pruebas de validación y autorización para todos los endpoints de `case`.
2.  **[BLOQUEANTE] Pruebas de autenticación y autorización:** Asegurar que todos los endpoints devuelvan `401 Unauthorized` sin un token válido.
3.  **[Alto Valor] Pruebas de Modo de Fallo:** Simular errores en la base de datos y en la generación de URLs SAS para garantizar que la API responda con un error `500` predecible y no se bloquee.
4.  **[Alto Valor] Lógica de `isPrimary`:** Validar explícitamente que al establecer una nueva imagen como primaria, todas las demás imágenes para esa misma entidad se marquen como no primarias.
5.  **[Recomendado] Pruebas Unitarias para Helpers:** Añadir pruebas para `safeName` y `blobExt` para aislar su lógica y prevenir bugs sutiles en el nombrado de archivos y la determinación de extensiones.
6.  **[Recomendado] Validación de `blobPath`:** Probar que la API rechaza o gestiona correctamente un intento de asociar un `blobPath` de una entidad (p.ej. `cases/...`) a una entidad de otro tipo (p.ej. un `pet`).

## 4. Estrategia de Mocking

La estrategia actual de usar `vi.hoisted` y `vi.mock` es excelente y debe continuar. Para implementar las pruebas de modo de fallo, extenderemos este patrón:

-   **Simular Fallo de Base de Datos:** Para cualquier prueba que deba verificar el manejo de errores de la BD, se usará:
    ```javascript
    mockQuery.mockRejectedValue(new Error('Database connection failed'));
    ```
-   **Simular Fallo de Generación de SAS URL:** Para probar el `catch` en la generación de URLs:
    ```javascript
    import * as blobStorage from '../server/blobStorage.js';
    vi.spyOn(blobStorage, 'generateUploadSasUrl').mockImplementation(() => {
      throw new Error('SAS generation failed');
    });
    ```
-   **Simular Fallo al Borrar Blob:** Para verificar que la API maneja con gracia un fallo en `deleteBlob`:
    ```javascript
    mockDeleteBlob.mockRejectedValue(new Error('Azure permissions error'));
    ```

## 6. Orden de Ejecución

Se recomienda el siguiente orden para abordar la implementación de las pruebas:

1.  **Prioridad 1 (Bloqueantes):** Implementar todas las pruebas para la **API de Imágenes de `case`**. Esto cierra la brecha de cobertura más crítica. Comenzar con `DELETE`, luego `POST`, `GET`, y las pruebas de autorización (`401`, `404`).
2.  **Prioridad 2 (Modo de Fallo):** Añadir las pruebas de manejo de errores para fallos de la base de datos y de la generación de SAS. Estos aseguran la robustez del servicio.
3.  **Prioridad 3 (Integridad de Datos):** Implementar la prueba de la lógica `isPrimary` para `pet` y `case`. Esto previene bugs visuales importantes en el frontend.
4.  **Prioridad 4 (Unitarias):** Crear el archivo de pruebas unitarias para `safeName` y `blobExt`. Son victorias rápidas que mejoran la confianza en el código.
5.  **Prioridad 5 (Refinamiento):** Implementar las pruebas restantes de menor prioridad, como la validación de `blobPath` cruzado y el manejo de errores en `deleteBlob`.
