# Plan de Pruebas: `media_boundary`

Este documento detalla el plan de pruebas para la superficie `media_boundary`, que gestiona la creación de URLs de almacenamiento, la persistencia de referencias de imágenes y las APIs relacionadas. El objetivo es identificar y priorizar las pruebas que faltan para mejorar la robustez, seguridad y fiabilidad del sistema.

## 1. Evaluación de Cobertura

El análisis de `tests/images.test.ts` muestra una base de pruebas de integración a nivel de ruta (route-level) que es un buen comienzo. Sin embargo, la cobertura actual presenta brechas significativas:

*   **Cobertura Parcial de Entidades:** Las pruebas existentes se centran principalmente en las rutas de `/api/pets/:id/images`. Las rutas análogas para `/api/cases/:id/images` están muy poco probadas, faltando por completo la cobertura para el endpoint `DELETE`.
*   **Falta de Pruebas Unitarias:** Las funciones auxiliares críticas como `safeName`, `blobExt`, y `generateId` en `images.ts` no tienen pruebas unitarias dedicadas. Su lógica se prueba solo de forma indirecta, lo que dificulta la depuración y el refactoring.
*   **Cobertura de Modos de Fallo Insuficiente:** Los tests actuales cubren los "caminos felices" y algunos errores de validación (404, 400). Sin embargo, no se prueba la respuesta del sistema a fallos de la base de datos (que deberían devolver un `500`) o fallos en la capa de `blobStorage`. Por ejemplo, el `catch(() => {})` en la ruta `DELETE` oculta un posible error de borrado en el almacenamiento, y este comportamiento no está probado.
*   **Lógica de Negocio no Verificada:** La lógica clave, como desmarcar una imagen como primaria (`is_primary`) cuando se sube una nueva imagen primaria, no se verifica explícitamente en las pruebas existentes. Solo se comprueba el estado de la nueva imagen, no el efecto colateral en las antiguas.
*   **Falta de Pruebas para `blobStorage.ts`:** No existen pruebas para el módulo `blobStorage.ts`. Aunque se mockea en los tests de las rutas, su propia lógica (ej. permisos correctos en las URLs SAS) no está verificada.

## 2. Pruebas Faltantes de Mayor Valor

Se priorizan las siguientes pruebas para maximizar el impacto en la calidad y estabilidad del código.

### Bloqueantes (Blockers)

Estas pruebas deben implementarse antes de cualquier nuevo despliegue de esta superficie, ya que cubren funcionalidades críticas y riesgos de regresión.

1.  **Paridad de Pruebas para `cases`:** Alcanzar la misma cobertura de pruebas para las rutas de `cases` que la que existe para `pets`. Esto incluye `POST`, `GET`, `DELETE` y todas las comprobaciones de autorización.
2.  **Prueba de `DELETE /api/cases/:id/images/:imageId`:** Añadir cobertura completa para este endpoint, que actualmente no tiene ninguna prueba.
3.  **Pruebas Unitarias para Helpers:** Crear pruebas unitarias para `safeName` y `blobExt`. Son rápidas de implementar y aseguran la correcta manipulación de nombres de archivo y extensiones.
4.  **Pruebas de Fallo de Base de Datos:** Verificar que todos los endpoints que interactúan con la base de datos devuelven un estado `500` con un error `db_error` cuando la función `query` lanza una excepción.
5.  **Verificación de Lógica `isPrimary`:** Añadir una prueba que verifique que al establecer una nueva imagen como primaria, cualquier otra imagen que fuera primaria para esa misma entidad se actualiza a `is_primary = FALSE`.
6.  **Validación de Propiedad para `upload-url` de `case`:** Añadir una prueba que confirme que el endpoint `POST /api/storage/upload-url` valida que el usuario es propietario del `case` cuando se proporcionan `entityType: 'case'` y `entityId`.

### Deseables (Nice-to-have)

Estas pruebas añaden capas adicionales de confianza y deben ser consideradas en futuros sprints.

1.  **Pruebas Unitarias para `blobStorage.ts`:** Probar que las funciones `generate...SasUrl` construyen las URLs con los permisos y tiempos de expiración correctos, mockeando el SDK de Azure.
2.  **Prueba de Fallo en `deleteBlob`:** Probar el comportamiento de los endpoints `DELETE` cuando `deleteBlob` falla. El código actual ignora el error; la prueba documentaría este comportamiento y facilitaría la discusión sobre si es el comportamiento deseado.
3.  **Prueba de `sort_order`:** Verificar que el `sort_order` se asigna correctamente de forma incremental al añadir múltiples imágenes a una misma entidad.

### Pruebas Unitarias (para `server/routes/images.ts`)

```typescript
// En un nuevo archivo, p.ej. tests/helpers.test.ts
describe('Unit tests for image route helpers', () => {

  describe('safeName()', () => {
    it('should replace special characters with underscores', () => {
      // Arrange: nombre con caracteres inválidos
      const unsafe = "file/with$special'chars.jpg";
      // Act: llamar a safeName
      const result = safeName(unsafe);
      // Assert: verificar que el resultado es seguro
      expect(result).toBe("file_with_special_chars.jpg");
    });

    it('should truncate long filenames', () => {
      // Arrange: nombre de archivo muy largo
      const longName = "a".repeat(100) + ".png";
      // Act: llamar a safeName
      const result = safeName(longName);
      // Assert: verificar longitud máxima de 80 caracteres
      expect(result.length).toBe(80);
    });
  });

  describe('blobExt()', () => {
    it('should extract extension from filename', () => {
      // Arrange: nombre de archivo con extensión
      // Act: llamar a blobExt
      // Assert: verificar que la extensión es correcta
      expect(blobExt('image.JPEG', 'application/octet-stream')).toBe('jpeg');
    });

    it('should infer extension from mimeType if filename has no extension', () => {
      // Arrange: sin extensión, pero con mimeType
      // Act: llamar a blobExt
      // Assert: verificar que se usa el mimeType
      expect(blobExt('image', 'image/png')).toBe('png');
    });

    it('should return "bin" for unknown types', () => {
      // Arrange: tipos desconocidos
      // Act: llamar a blobExt
      // Assert: verificar que el fallback es 'bin'
      expect(blobExt('document', 'application/zip')).toBe('bin');
    });
  });
});
```

## 4. Estrategia de Mocking

*   **Pruebas de Ruta (`images.ts`):** Continuar con la estrategia actual de mockear los módulos `db.js`, `firebaseAdmin.js`, y `blobStorage.js`. Para probar modos de fallo, configurar los mocks para que fallen:
    *   `mockQuery.mockRejectedValue(new Error('DB Error'))` para simular un fallo de base de datos.
    *   `mockDeleteBlob.mockRejectedValue(new Error('Storage Error'))` para simular un fallo de borrado de blob.
*   **Pruebas de Lógica de Negocio:** Para verificar la lógica de `isPrimary`, se debe usar `mockQuery.toHaveBeenCalledWith(...)` para asegurar que la consulta `UPDATE` correcta fue enviada a la base de datos con los parámetros adecuados.
*   **Pruebas Unitarias (`blobStorage.ts`):** Mockear el módulo `@azure/storage-blob` para interceptar las llamadas a `generateBlobSASQueryParameters` y `BlobServiceClient`. Esto permite verificar los parámetros (como `permissions`) sin realizar llamadas de red reales.
*   **Pruebas Unitarias (Helpers):** No se necesitan mocks. Son funciones puras que se pueden probar con entradas y salidas directas.

## 6. Orden de Ejecución

Se recomienda seguir este orden para implementar las pruebas:

1.  **Implementar Pruebas Unitarias para Helpers:** Empezar con `safeName` y `blobExt`. Son victorias rápidas que no dependen de mocks complejos.
2.  **Alcanzar Paridad en `Case Images API`:** Copiar y adaptar las pruebas existentes de `Pet Images API` para cubrir los endpoints de `Case Images API`, incluyendo el `DELETE` faltante y la autorización. **Esta es la máxima prioridad** para evitar regresiones.
3.  **Implementar Pruebas de Modo de Fallo:** Añadir las pruebas que verifican el comportamiento de los endpoints cuando la base de datos falla (respuesta `500`).
4.  **Implementar Prueba de Lógica de Negocio `isPrimary`:** Añadir la prueba crucial que verifica la degradación de la imagen primaria anterior.
5.  **Implementar Pruebas Deseables:** Si el tiempo lo permite, abordar las pruebas para `blobStorage.ts` y otros casos de borde como el fallo en `deleteBlob`.
