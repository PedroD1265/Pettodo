Claro, aquí está la auditoría de riesgo para la superficie `media_boundary`.

### 1. Resumen Ejecutivo

La superficie `media_boundary` gestiona la creación de URLs de carga, la persistencia de referencias a imágenes y las operaciones CRUD para imágenes asociadas a mascotas (`pets`) y casos (`cases`). La auditoría revela una vulnerabilidad **Crítica** de seguridad en la forma en que el sistema confía en las rutas de los blobs (`blobPath`) proporcionadas por el cliente. Esto permite a un usuario asociar una imagen con una entidad incorrecta a la que tiene acceso, lo que podría llevar a la corrupción de datos.

Además, se identificaron riesgos **Altos** relacionados con la integridad de los datos debido a la falta de transacciones en la base de datos, lo que podría dejar la base de datos en un estado inconsistente si una operación falla a la mitad. También se encontraron riesgos **Medios** de fragilidad operacional, como la creación de archivos huérfanos en el almacenamiento y una duplicación de código significativa que aumenta la carga de mantenimiento y el riesgo de errores.

La superficie se beneficiaría enormemente de una validación más estricta en el backend, el uso de transacciones de base de datos y la refactorización para eliminar la lógica duplicada.

### 2. Principales Riesgos

*   **Crítico: Confianza indebida en el `blobPath` proporcionado por el cliente.**
    Al asociar una imagen a una entidad (mascota o caso), el endpoint `POST /api/pets/:id/images` acepta un `blobPath` del cuerpo de la solicitud. El servidor verifica que el usuario es propietario del `petId` en la URL, pero no valida que el `blobPath` corresponda a esa misma entidad. Un usuario malintencionado podría obtener una URL de carga para su `pet-A`, subir una imagen, y luego usar ese `blobPath` para asociarla con su `pet-B`, corrompiendo los datos o enlazando imágenes de manera incorrecta.

*   **Alto: Ausencia de Transacciones en Base de Datos.**
    Al crear una nueva referencia de imagen (`POST .../images`), se realizan múltiples operaciones de base de datos secuenciales: una para desmarcar la imagen primaria anterior (`UPDATE`), otra para obtener el siguiente `sort_order` (`SELECT`), y una final para insertar el nuevo registro (`INSERT`). Estas operaciones no están envueltas en una transacción atómica. Si la operación `INSERT` falla después de que el `UPDATE` haya tenido éxito, la entidad podría quedar sin una imagen primaria, resultando en un estado de datos inconsistente.

*   **Medio: Creación de Archivos Huérfanos en el Almacenamiento.**
    El endpoint para eliminar una imagen (`DELETE .../images/:imageId`) borra primero el registro de la base de datos y luego intenta eliminar el blob de Azure Storage. La llamada para eliminar el blob (`deleteBlob`) tiene un `.catch(() => {})` que suprime silenciosamente cualquier error. Si la eliminación del blob falla (por ejemplo, debido a un problema de red transitorio), el registro de la base de datos desaparecerá, pero el archivo permanecerá en el almacenamiento de forma indefinida como un "huérfano". Esto genera costos de almacenamiento innecesarios y deja datos que deberían haber sido eliminados.

*   **Medio: Duplicación Extensa de Código.**
    La lógica para manejar las imágenes de `pets` y `cases` en `server/routes/images.ts` es casi idéntica. Los endpoints `POST`, `GET` y `DELETE` para ambas entidades siguen el mismo patrón. Esta duplicación aumenta el esfuerzo de mantenimiento y crea el riesgo de que se corrija un error en una implementación pero se olvide en la otra, lo que conduce a una deriva arquitectónica y a un comportamiento inconsistente.

### 3. Evidencia del Paquete

*   **Riesgo de Confianza en `blobPath` (Crítico):**
    En `server/routes/images.ts`, el endpoint para añadir una imagen a una mascota confía plenamente en el `blobPath` del cliente.
    ```typescript
    // server/routes/images.ts:114
    router.post('/pets/:id/images', verifyToken, async (req: AuthenticatedRequest, res: Response) => {
      // ... verificación de propiedad de petId ...
    
      const { blobPath, ... } = req.body as { blobPath?: string; ... }; // blobPath viene del cliente
    
      if (!blobPath) { // solo valida que exista
        res.status(400).json({ error: 'validation', message: 'blobPath is required' });
        return;
      }
    
      // ...
      await query(
        `INSERT INTO pet_images (..., blob_path, ...) VALUES (..., $4, ...)`,
        [..., blobPath, ...], // El blobPath se inserta sin validar su correspondencia con petId
      );
      // ...
    });
    ```

*   **Ausencia de Transacciones (Alto):**
    La secuencia de operaciones de base de datos no está dentro de un bloque `BEGIN`/`COMMIT`.
    ```typescript
    // server/routes/images.ts:133-143
    try {
      if (isPrimary) {
        // Operación 1: UPDATE
        await query('UPDATE pet_images SET is_primary = FALSE WHERE pet_id = $1', [petId]);
      }
      // Operación 2: SELECT
      const sortRes = await query('SELECT COALESCE(MAX(sort_order), -1) + 1 AS next FROM pet_images WHERE pet_id = $1', [petId]);
      const sortOrder = sortRes.rows[0].next ?? 0;

      // Operación 3: INSERT
      await query(
        `INSERT INTO pet_images ...`,
        [...]
      );
      // No hay transacción que envuelva estas tres operaciones.
    } // ...
    ```

*   **Manejo de Errores en Eliminación (Medio):**
    Los errores al eliminar el blob de almacenamiento son ignorados.
    ```typescript
    // server/routes/images.ts:186-188
    if (isStorageConfigured()) {
      // Si deleteBlob falla, el error es capturado y descartado.
      await deleteBlob(blob_path).catch(() => {});
    }
    ```

*   **Duplicación de Código (Medio):**
    El fichero `server/routes/images.ts` contiene una sección para `Pet image references` (líneas 107-192) y otra para `Case image references` (líneas 196-281) cuya lógica es prácticamente un espejo. Por ejemplo, `POST /api/pets/:id/images` y `POST /api/cases/:id/images` son casi idénticos.

### 4. Mitigaciones Sugeridas

1.  **Validar la correspondencia del `blobPath`:** En los endpoints `POST .../images`, antes de insertar en la base de datos, el servidor debe validar que el `blobPath` proporcionado por el cliente corresponde a la entidad que se está modificando. Por ejemplo, para `POST /api/pets/pet-123/images`, el `blobPath` debe comenzar con el prefijo esperado, como `pets/pet-123/`.
    ```typescript
    // En POST /api/pets/:id/images
    const expectedPrefix = `pets/${petId}/`;
    if (!blobPath.startsWith(expectedPrefix)) {
      return res.status(400).json({ error: 'validation', message: 'blobPath does not match entity' });
    }
    ```

2.  **Usar Transacciones de Base de Datos:** Envuelva las operaciones de base de datos (`UPDATE`, `SELECT`, `INSERT`) en una transacción. La mayoría de los clientes de bases de datos (como `node-postgres`) permiten esto obteniendo un cliente del pool y usando `BEGIN`, `COMMIT` y `ROLLBACK`.

3.  **Mejorar el Manejo de Errores de Eliminación:** En lugar de ignorar los errores de `deleteBlob`, regístrelos. Para una solución más robusta, considere implementar un mecanismo de reintento o mover las eliminaciones fallidas a una cola de trabajos para su procesamiento posterior, asegurando que los archivos huérfanos se limpien eventualmente.

4.  **Refactorizar la Lógica Duplicada:** Cree una función o un middleware genérico que maneje la lógica de asociación de imágenes. Podría aceptar el tipo de entidad (`pet`, `case`), el ID de la entidad y los datos de la imagen, y luego realizar las verificaciones y operaciones de base de datos necesarias, reduciendo así la duplicación.

### 5. Victorias Rápidas (Quick Wins)

1.  **Añadir validación de `blobPath`:** Implementar la comprobación del prefijo del `blobPath` como se sugiere en la mitigación #1. Este es un cambio pequeño y de alto impacto para la seguridad.
2.  **Registrar errores de eliminación:** Cambie `.catch(() => {})` a `.catch(err => console.error('Failed to delete blob:', blob_path, err))`. Esto al menos proporcionará visibilidad sobre los fallos sin necesidad de un sistema complejo de reintentos.
3.  **Registrar errores de base de datos:** En todos los bloques `catch` genéricos, añada un registro del error real antes de enviar la respuesta 500.
    ```typescript
    // Ejemplo
    catch (err: any) {
      console.error(`DB Error in ${req.path}:`, err); // Añadir esto
      res.status(500).json({ error: 'db_error' }); return;
    }
    ```
