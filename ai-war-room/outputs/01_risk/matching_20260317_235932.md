# Auditoría de Riesgo de PETTODO

## 1. Resumen Ejecutivo

La superficie `matching` es funcional en su estado actual, pero representa una versión muy simplificada y frágil del sistema previsto en la documentación (`AI_MATCHING.md`). La implementación actual es puramente heurística (basada en reglas simples) y no utiliza ninguna capacidad de IA, lo que crea una **deriva arquitectónica crítica** y establece expectativas incorrectas.

Los riesgos más significativos identificados son:

1.  **Lógica de negocio duplicada y divergente** entre el backend y el frontend, lo que garantiza inconsistencias y un alto coste de mantenimiento.
2.  Un **error de cálculo fundamental** en la puntuación de `confidence` (confianza), que socava la fiabilidad del resultado principal de la funcionalidad.
3.  **Manejo de datos frágil** que puede llevar a fallos silenciosos y puntuaciones incorrectas sin generar errores.
4.  **Riesgos operativos** de rendimiento y posible denegación de servicio (DoS) debido a un procesamiento de datos ineficiente.

Aunque la ruta está protegida por autenticación, su diseño actual como un "MVP" (Producto Mínimo Viable) introduce riesgos significativos de corrección, mantenibilidad y confianza del producto. La brecha entre el diseño y la implementación es la mayor amenaza para la superficie.

## 2. Riesgos Principales

### **CRÍTICO: Deriva Arquitectónica y Desalineación con la Documentación**

-   **Descripción:** La implementación en `server/routes/matching.ts` es un sistema de puntuación heurístico simple que se ejecuta en memoria. Esto contradice directamente el diseño detallado y multi-etapa descrito en `docs/02_build/AI_MATCHING.md`, que prevé un pipeline de IA con puertas de calidad de imagen, generación de embeddings y fases de recuperación de candidatos. El nombre "AI_MATCHING" es, en este momento, aspiracional y engañoso.
-   **Impacto:** Erosiona la confianza en la documentación, crea deuda técnica y establece expectativas falsas para el producto. El sistema actual no está preparado para evolucionar hacia la visión documentada sin una reescritura importante.
-   **Categorías:** Architectural Drift, High-risk assumptions.

### **CRÍTICO: Lógica de Puntuación Duplicada y Divergente**

-   **Descripción:** Existe una implementación casi idéntica, pero sutilmente diferente, de la lógica de puntuación en el backend (`server/routes/matching.ts`) y en el frontend (`src/app/utils/matching.ts`). Por ejemplo, el backend limita la puntuación de colores y rasgos (`Math.min(...)`), mientras que el frontend no lo hace.
-   **Impacto:** Esto es una fuente garantizada de bugs. Cualquier cambio en la lógica de puntuación debe hacerse en dos lugares, y es probable que diverjan aún más con el tiempo. Dado que el cliente real (`matchingRealAdapter`) llama a la API del backend, el código del frontend es probablemente código muerto o se usa en un contexto no obvio, lo que aumenta la confusión.
-   **Categorías:** Correctness and bug risk, Operational fragility.

### **ALTO: Cálculo Incorrecto de la Puntuación de Confianza**

-   **Descripción:** La puntuación de `confidence` se calcula como `Math.min(Math.round((score / 95) * 100), 95)`. El denominador `95` pretende ser la puntuación máxima posible. Sin embargo, un cálculo de la puntuación máxima real en el backend es: `distance (30) + recency (30) + size (20) + colors (20) + traits (15) = 115`.
-   **Impacto:** El denominador incorrecto infla artificialmente la puntuación de confianza y la lógica es fundamentalmente errónea. Una puntuación de `score = 95` da como resultado `confidence = 95` (debido al `Math.min`), pero una puntuación de `score = 115` también da como resultado `confidence = 95`. Esto oculta la verdadera calidad de la coincidencia y hace que la métrica de `confidence` sea poco fiable.
-   **Categorías:** Correctness and bug risk.

### **MEDIO: Falta de Validación Robusta de Entradas y Fallos Silenciosos**

-   **Descripción:** El código utiliza `parseFloat(origin.approx_lat)` y `Number(c.created_at)` sin verificar si el resultado es `NaN` (Not a Number). Si los datos en la base de datos no son numéricos por alguna razón (p. ej., una cadena vacía o corrupta), estas funciones devolverán `NaN`. Las operaciones posteriores (cálculos de distancia, tiempo) fallarán silenciosamente, resultando en una puntuación de `0` para esa categoría sin ningún error registrado.
-   **Impacto:** Las coincidencias pueden recibir puntuaciones incorrectamente bajas debido a datos mal formados, lo que degrada la calidad del matching sin que los desarrolladores o usuarios se den cuenta del problema subyacente.
-   **Categorías:** Data handling and validation, Bug risk.

### **MEDIO: Potencial de Degradación del Rendimiento o Denegación de Servicio (DoS)**

-   **Descripción:** El endpoint recupera hasta 300 casos completos de la base de datos (`LIMIT 300`) y luego itera sobre ellos en memoria en el servidor de Node.js. Para un caso origen con muchos candidatos potenciales, esta operación puede consumir una cantidad significativa de CPU y memoria.
-   **Impacto:** Un usuario malintencionado (o incluso un uso legítimo intensivo) podría desencadenar repetidamente esta costosa operación, degradando el rendimiento del servicio para todos los usuarios o incluso causando una denegación de servicio. La consulta no es selectiva en los campos, cargando datos innecesarios.
-   **Categorías:** Operational fragility.

## 3. Evidencia del Paquete

-   **Deriva Arquitectónica:** `docs/02_build/AI_MATCHING.md` describe un pipeline complejo (secciones 8.1 a 8.8) que incluye "Stage B — Cheap quality gate", "Stage E — Feature and embedding generation", etc. La implementación en `server/routes/matching.ts` es un único bucle `for` que aplica heurísticas simples, sin ninguna interacción con imágenes o modelos de IA.

-   **Lógica Duplicada:**
    -   Backend (`server/routes/matching.ts`, línea 132): `score += Math.min(colorMatches * 10, 20);`
    -   Frontend (`src/app/utils/matching.ts`, línea 77): `score += colorMatches * 10;` (falta el límite de 20).
    -   Backend (línea 144): `score += Math.min(sharedTraits.length * 5, 15);`
    -   Frontend (línea 83): `score += sharedTraits.length * 5;` (falta el límite de 15).

-   **Error en Cálculo de Confianza:**
    -   `server/routes/matching.ts`, línea 148: `const confidence = Math.min(Math.round((score / 95) * 100), 95);`
    -   La suma de las puntuaciones máximas posibles de las funciones `...Score` y las reglas en línea es 115, no 95.

-   **Fallos Silenciosos:**
    -   `server/routes/matching.ts`, líneas 91-92: `const originLat = origin.approx_lat != null ? parseFloat(origin.approx_lat) : null;`. No hay una comprobación posterior con `isNaN(originLat)`.
    -   `server/routes/matching.ts`, línea 112: `const r = recencyScore(Number(c.created_at));`. No hay comprobación con `isNaN` para la conversión `Number()`.

-   **Rendimiento/DoS:**
    -   `server/routes/matching.ts`, líneas 82-87: La consulta `SELECT * FROM cases ... LIMIT 300` carga todas las columnas para 300 filas, que luego se procesan en un bucle síncrono (`for...of`) en el hilo de Node.js.

-   **Interfaz Inconsistente:**
    -   El backend en `server/routes/matching.ts` devuelve un objeto con `caseId`, `confidence`, `reasons`, etc.
    -   La interfaz `MatchResult` en `src/app/utils/matching.ts` espera `{ case: Case; confidence: number; reasons: string[] }`. El cliente (`api.ts`) probablemente espera este formato, lo que causará un error de tipado o de tiempo de ejecución al intentar acceder a `result.case.id` cuando solo existe `result.caseId`.

## 4. Mitigaciones Sugeridas

1.  **Resolver la Deriva Arquitectónica:** Tomar una decisión explícita:
    a.  **Aceptar el MVP:** Actualizar `AI_MATCHING.md` para reflejar la realidad heurística actual, documentando esto como "Fase 1" y moviendo el pipeline de IA a un documento de "hoja de ruta futura".
    b.  **Construir hacia la Visión:** Crear un plan por etapas para implementar el pipeline documentado, comenzando con la extracción de la lógica de puntuación a una capa de servicio y la adición de una puerta de calidad de imagen básica.

2.  **Eliminar la Lógica Duplicada:** El frontend debe confiar exclusivamente en el backend para la lógica de negocio.
    -   Eliminar el fichero `src/app/utils/matching.ts` por completo para evitar su uso accidental.
    -   Asegurar que todas las vistas que necesiten mostrar coincidencias utilicen el `matchingRealAdapter` que llama a la API del backend.

3.  **Corregir el Cálculo de Confianza:**
    -   Definir una constante `MAX_POSSIBLE_SCORE` en `server/routes/matching.ts` con el valor correcto (115).
    -   Actualizar la fórmula: `const confidence = Math.round((score / MAX_POSSIBLE_SCORE) * 100);`.
    -   Reevaluar si el tope artificial de `95` en `Math.min` tiene un propósito de producto; si no, eliminarlo para que la confianza pueda llegar a 100.

4.  **Hacer el Manejo de Datos más Robusto:**
    -   Después de cada `parseFloat` o `Number`, añadir una comprobación `if (isNaN(variable))`. Si es `NaN`, se debe registrar una advertencia (`console.warn`) y omitir esa parte de la puntuación de forma explícita, en lugar de fallar silenciosamente.

5.  **Mejorar el Rendimiento y la Resiliencia:**
    -   Modificar la consulta SQL para seleccionar solo las columnas necesarias para la puntuación (p. ej., `id`, `approx_lat`, `approx_lng`, `created_at`, `size`, `colors`, `traits`).
    -   Considerar mover el cálculo de distancia a la base de datos (p. ej., con PostGIS si se usa PostgreSQL), lo que permite ordenar por distancia y reducir el conjunto de candidatos de manera más eficiente.
    -   Implementar limitación de velocidad (rate limiting) en el endpoint `/api/matching/cases/:id` para prevenir abusos.

## 5. Victorias Rápidas

1.  **Corregir el denominador de la confianza:** Cambiar `95` por `115` (o el valor correcto tras una revisión) en `server/routes/matching.ts`. Este es un cambio de una línea que corrige un error fundamental.
2.  **Añadir validaciones `isNaN`:** Agregar las comprobaciones para `lat`, `lng` y `created_at` para prevenir fallos silenciosos.
3.  **Eliminar el código muerto:** Si se confirma que `src/app/utils/matching.ts` no se utiliza, eliminarlo del repositorio para reducir la confusión.
4.  **Corregir la interfaz del cliente:** Ajustar el tipo de dato que `matchingApi.run` espera en `src/app/services/api.ts` para que coincida con la respuesta real del backend, o ajustar el backend para que devuelva la forma esperada.
