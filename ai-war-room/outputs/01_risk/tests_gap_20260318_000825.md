# Auditoría de Riesgos de PETTODO

## 1. Resumen Ejecutivo

La auditoría de la superficie `tests_gap` revela una brecha crítica entre la funcionalidad implementada en el backend y la cobertura de pruebas automatizadas. Funcionalidades centrales y de alto riesgo, incluido el algoritmo principal de coincidencia de mascotas (`matching`) y varias rutas sensibles como el contacto protegido y el reporte de abusos, carecen por completo de pruebas.

El riesgo más grave es la ausencia total de pruebas para `server/routes/matching.ts`. Este componente crítico contiene una lógica de puntuación compleja y propensa a errores que actualmente opera sin ninguna validación automatizada, poniendo en riesgo la fiabilidad de una de las características distintivas del producto. Además, se identificó un probable bug en el cálculo del puntaje de confianza.

También se han encontrado otros riesgos significativos, como una vulnerabilidad en el manejo de imágenes que podría permitir la asociación de datos incorrectos, y una divergencia importante entre la arquitectura documentada para el matching "IA" y su implementación real, que es mucho más simple.

La falta de cobertura de pruebas para numerosas rutas sensibles (`protected-contact`, `abuse`, `reviews`, etc.) es alarmante y debe ser abordada con la máxima prioridad para garantizar la seguridad, corrección y estabilidad operativa de la plataforma.

## 3. Evidencia del Paquete

- **Falta de Archivo de Pruebas para Matching**: El paquete de auditoría contiene pruebas para `auth`, `cases`, `images`, `import`, `pets` y `public`. Sin embargo, no existe un archivo `tests/matching.test.ts` que corresponda a la ruta crítica `server/routes/matching.ts`.
- **Rutas sin Cobertura de Pruebas**: El archivo `server/app.ts` monta los siguientes enrutadores para los cuales no se proporcionaron archivos de prueba: `protectedContactRoutes`, `communityDogRoutes`, `changeRequestRoutes`, `evidenceRoutes`, `reviewRoutes`, y `abuseRoutes`.
- **Lógica de Puntuación no Verificada**: En `server/routes/matching.ts`, la confianza se calcula con la línea:
  ```typescript
  const confidence = Math.min(Math.round((score / 95) * 100), 95);
  ```
  La suma máxima de los componentes de puntuación parece ser 115 (30 por distancia + 30 por recencia + 20 por tamaño + 20 por colores + 15 por rasgos). El uso del divisor `95` no está documentado y es probablemente un error que subestima o sobreestima la confianza.
- **`blobPath` Confiado al Cliente**: En `server/routes/images.ts`, el endpoint `POST /api/pets/:id/images` recibe el `blobPath` del `req.body` y lo inserta en la base de datos sin validar su formato o prefijo. Solo se verifica que la mascota (`petId`) pertenezca al usuario.
  ```typescript
  // server/routes/images.ts:
  const { blobPath, ... } = req.body as { ... };
  // ...
  if (!blobPath) { // solo valida existencia, no formato o pertenencia
    res.status(400).json({ error: 'validation', message: 'blobPath is required' });
    return;
  }
  // ...
  await query(
    `INSERT INTO pet_images (..., blob_path, ...) VALUES (..., $4, ...)`,
    [..., blobPath, ...],
  );
  ```
- **Deriva Arquitectónica**: El documento `docs/02_build/AI_MATCHING.md` describe un pipeline sofisticado con "Stage E — Feature and embedding generation". La implementación en `server/routes/matching.ts` no utiliza embeddings ni IA, sino funciones heurísticas como `haversineKm` y `colorOverlap`.

## 4. Mitigaciones Sugeridas

1.  **Crear una Suite de Pruebas para `matching.ts` (Crítico)**:
    -   Escribir pruebas unitarias para las funciones auxiliares de puntuación (`recencyScore`, `distanceScore`, etc.) para verificar su lógica de forma aislada.
    -   Desarrollar pruebas de integración para el endpoint `GET /api/matching/cases/:id`. Usar datos de casos simulados que cubran varios escenarios: valores nulos en campos clave (`approx_lat`, `colors`, etc.), casos sin rasgos en común y casos con alta similitud para validar que el ranking y los puntajes son correctos y predecibles.
    -   Verificar el comportamiento del umbral de confianza (`confidence < 15`).

2.  **Implementar Pruebas para Rutas Faltantes (Crítico)**:
    -   Crear archivos de prueba para las rutas actualmente sin cobertura, priorizando por nivel de riesgo:
        1.  `abuseRoutes` y `protectedContactRoutes` (riesgo de seguridad y confianza).
        2.  `reviewRoutes` y `changeRequestRoutes` (riesgo de integridad de datos).
        3.  `evidenceRoutes` y `communityDogRoutes` (riesgo de funcionalidad central).

3.  **Validar el `blobPath` en el Servidor (Alto)**:
    -   Modificar los endpoints `POST /api/pets/:id/images` y `POST /api/cases/:id/images` en `server/routes/images.ts`. Antes de insertar el `blobPath` en la base de datos, el servidor debe validar que la ruta comienza con un prefijo esperado y seguro, como `temp/{uid}/`. Cualquier ruta que no coincida con el patrón esperado debe ser rechazada con un error 400.

4.  **Resolver la Deriva Arquitectónica (Alto)**:
    -   El equipo de producto y arquitectura debe decidir si la implementación actual del matching es un MVP intencional o una desviación.
    -   **Opción A**: Si la implementación simple es la deseada por ahora, el documento `docs/02_build/AI_MATCHING.md` debe ser actualizado para reflejar la realidad del sistema heurístico, eliminando las referencias a "IA" y "embeddings" para evitar confusiones.
    -   **Opción B**: Si la visión de IA sigue siendo el objetivo, se deben crear historias de usuario o tareas técnicas para cerrar la brecha entre la implementación actual y la arquitectura documentada.
