## 1. Resumen Ejecutivo

La auditoría de la superficie `tests_gap` revela una brecha crítica entre la funcionalidad implementada y la cobertura de pruebas automatizadas. El riesgo más significativo es la **ausencia total de pruebas para la ruta de coincidencias (`/api/matching/cases/:id`)**, un componente central y complejo del sistema. Esta ruta utiliza una lógica heurística no validada para clasificar posibles coincidencias de mascotas, lo que introduce un alto riesgo de errores, resultados incorrectos e inestabilidad.

Además, el análisis del `server/app.ts` muestra que numerosas rutas de API con implicaciones de seguridad y confianza (como `protected-contact`, `abuse`, `reviews`) carecen por completo de archivos de prueba, dejando funcionalidades críticas sin ninguna validación automatizada. También se ha identificado una notable **desviación arquitectónica** entre la sofisticada estrategia de "AI Matching" documentada y la implementación heurística mucho más simple que existe actualmente, lo que puede generar expectativas desalineadas en el producto.

Finalmente, se observan fragilidades operativas, como la supresión silenciosa de errores durante la eliminación de archivos en el almacenamiento de blobs, lo que podría llevar a la acumulación de datos huérfanos y costos de almacenamiento innecesarios.

**El paquete de auditoría está incompleto**, ya que faltan los archivos fuente de la mayoría de las rutas de la API. La evaluación completa del riesgo para esas superficies es imposible, pero su ausencia en el conjunto de pruebas es en sí misma un riesgo crítico.

## 2. Riesgos Principales

1.  **Severidad: Crítica** - **Ausencia total de pruebas para la lógica de coincidencias.**
    La ruta `server/routes/matching.ts` contiene una lógica de negocio compleja (fórmula de Haversine, múltiples funciones de puntuación heurística) que es fundamental para el producto. No existe un archivo `tests/matching.test.ts`, lo que significa que esta funcionalidad crítica no está validada. Errores en los cálculos, manejo de valores nulos o lógica de clasificación podrían pasar desapercibidos, degradando severamente la fiabilidad del producto.

2.  **Severidad: Crítica** - **Múltiples superficies de API de alto riesgo sin cobertura de pruebas.**
    El archivo `server/app.ts` registra numerosas rutas (`protected-contact`, `abuse`, `review`, `change-requests`, `community-dogs`, `evidence`) para las cuales no existen archivos de prueba correspondientes en el directorio `tests/`. Estas rutas, por sus nombres, gestionan áreas sensibles como el contacto protegido entre usuarios, reportes de abuso y flujos de revisión, representando un riesgo masivo de seguridad, privacidad y confianza si no se prueban adecuadamente.

3.  **Severidad: Alta** - **Desviación arquitectónica entre la visión y la implementación del "Matching".**
    El documento `docs/02_build/AI_MATCHING.md` describe un sistema de coincidencias multifase, sofisticado y consciente de la calidad. La implementación real en `server/routes/matching.ts` es un sistema de puntuación heurístico mucho más simple. Esta discrepancia representa un riesgo de expectativas desalineadas con el equipo de producto y los stakeholders, y oculta la verdadera madurez de la funcionalidad. Las pruebas (si existieran) solo validarían el modelo simple, no la visión documentada.

4.  **Severidad: Media** - **Lógica de puntuación frágil y no validada.**
    La lógica en `server/routes/matching.ts` depende de "números mágicos" (ej. `30`, `22`, `15`) sin justificación y de comparaciones de texto simplistas (`toLowerCase().includes()`). Esto hace que el sistema sea frágil y propenso a falsos positivos/negativos (ej. un rasgo 'shy' coincidiendo con 'bushy tail'). Sin pruebas, es imposible verificar la sensatez o el impacto de estos valores y lógicas.

5.  **Severidad: Media** - **Fragilidad operativa en el manejo de almacenamiento.**
    En `server/routes/images.ts`, las operaciones de eliminación de blobs (`deleteBlob`) capturan y silencian cualquier error (`.catch(() => {})`). Si la API de almacenamiento de Azure falla, el registro de la base de datos se elimina, pero el archivo permanece, convirtiéndose en un "archivo huérfano". Esto conduce a una degradación de la integridad de los datos, costos de almacenamiento desperdiciados y una operación frágil.

## 3. Evidencia del Paquete

-   **Riesgo 1 y 2 (Ausencia de pruebas críticas):**
    -   El archivo `server/app.ts` muestra el registro de rutas como `app.use("/api", matchingRoutes);`, `app.use("/api", abuseRoutes);`, y `app.use("/api", protectedContactRoutes);`.
    -   El listado del directorio `tests/` (`tests/auth.test.ts`, `tests/cases.test.ts`, etc.) demuestra la ausencia de `tests/matching.test.ts`, `tests/abuse.test.ts`, y otros archivos de prueba cruciales.
    -   `server/routes/matching.ts` contiene una lógica compleja sin pruebas, como la función `haversineKm` y las funciones de puntuación `recencyScore` y `distanceScore`.

-   **Riesgo 3 (Desviación arquitectónica):**
    -   `docs/02_build/AI_MATCHING.md` (Sección 8, "Matching pipeline direction") describe un proceso de múltiples etapas: "Stage A — Input", "Stage B — Cheap quality gate", "Stage E — Feature and embedding generation", etc.
    -   `server/routes/matching.ts`, en cambio, implementa un único bucle `for (const c of candidatesRes.rows)` que aplica directamente sumas de puntuaciones heurísticas, una arquitectura mucho más simple que la documentada.

-   **Riesgo 4 (Lógica de puntuación frágil):**
    -   En `server/routes/matching.ts`:
        ```typescript
        // "Números mágicos" sin contexto
        function distanceScore(km: number): { score: number; reason: string | null } {
          if (km < 0.5) return { score: 30, reason: 'very close to your area' };
          if (km < 1) return { score: 22, reason: 'near your area' };
          // ...
        }

        // Lógica de coincidencia de texto simplista
        const sharedTraits = originTraits.filter((t) =>
          cTraits.some(
            (ct) =>
              ct.toLowerCase().includes(t.toLowerCase()) ||
              t.toLowerCase().includes(ct.toLowerCase())
          )
        );
        ```

-   **Riesgo 5 (Fragilidad operativa):**
    -   En `server/routes/images.ts`, dentro de `DELETE /api/pets/:id/images/:imageId`:
        ```typescript
        if (isStorageConfigured()) {
          await deleteBlob(blob_path).catch(() => {}); // El error se ignora silenciosamente
        }
        ```
    -   El mismo patrón se repite en `DELETE /api/cases/:id/images/:imageId`.

## 4. Mitigaciones Sugeridas

1.  **Crear Pruebas para la Lógica de Coincidencias (Crítico):**
    -   Crear el archivo `tests/matching.test.ts`.
    -   Implementar pruebas unitarias para las funciones auxiliares (`haversineKm`, `recencyScore`, `distanceScore`, `colorOverlap`, etc.) con datos de entrada variados y casos límite.
    -   Escribir pruebas de integración para el endpoint `GET /api/matching/cases/:id` que simulen diferentes datos de "casos" y verifiquen que el ranking y las puntuaciones de confianza son lógicos y predecibles.

2.  **Establecer Cobertura de Pruebas Mínima para Rutas Faltantes:**
    -   Para cada archivo de ruta en `server/routes/` que no tenga un `*.test.ts`, crear un archivo de prueba esqueleto.
    -   Como mínimo, agregar una prueba de "happy path" (ej. que la ruta principal devuelve 200 con autenticación válida) y una prueba de autorización (devuelve 401 sin token). Esto establece la estructura y hace visible la brecha en los informes de cobertura. Priorizar `protected-contact`, `abuse` y `reviews`.

3.  **Alinear la Documentación con la Realidad:**
    -   Iniciar una conversación con el equipo de producto para aclarar si la implementación actual en `matching.ts` es el MVP intencionado.
    -   Si es así, actualizar `docs/02_build/AI_MATCHING.md` para reflejar la implementación heurística actual y marcar las secciones más avanzadas como "Futuro".
    -   Si no lo es, crear un plan de trabajo técnico para evolucionar la implementación actual hacia la visión documentada.

4.  **Refactorizar la Lógica de Puntuación:**
    -   Extraer los "números mágicos" de `matching.ts` a constantes con nombres descriptivos (ej. `VERY_RECENT_SIGHTING_SCORE = 30`).
    -   Mejorar la lógica de coincidencia de rasgos y colores para que sea más robusta, por ejemplo, normalizando y dividiendo las cadenas en "tokens" en lugar de usar `includes()`.

5.  **Mejorar el Manejo de Errores de Almacenamiento:**
    -   En `server/routes/images.ts`, modificar el manejo de errores de `deleteBlob`. Como mínimo, registrar el error de forma explícita para que sea visible (`console.error`).
    -   Para una solución más robusta, considerar una estrategia de reintentos o, si la eliminación no es crítica para el éxito de la solicitud, agregar la ruta del blob a una cola de "limpieza" para un procesamiento asíncrono.

## 5. Victorias Rápidas

1.  **Iniciar el archivo de pruebas de `matching`:** Crear `tests/matching.test.ts` y añadir una prueba simple que verifique que la ruta devuelve un `200` con un `caseId` válido y un `404` con uno inválido. Esto establece el andamiaje para pruebas más complejas y se puede hacer en minutos.

2.  **Hacer visibles los errores de eliminación de blobs:** Cambiar `deleteBlob(...).catch(() => {})` a `deleteBlob(...).catch(err => console.error('Failed to delete blob', blob_path, err))`. Esto no soluciona el problema de los archivos huérfanos, pero hace que el fallo sea inmediatamente visible en los logs del servidor.

3.  **Añadir comentarios a los "números mágicos":** En `server/routes/matching.ts`, añadir comentarios de código explicando brevemente el propósito de cada valor en las funciones de puntuación, como un primer paso antes de refactorizarlos a constantes.
