### 1. Resumen Ejecutivo

La superficie `media_boundary` gestiona la generación de URLs para subir archivos, la persistencia de referencias a imágenes y los endpoints de API relacionados para mascotas (`pets`) y casos (`cases`). La auditoría ha revelado varias vulnerabilidades críticas y riesgos importantes.

El riesgo más grave es una vulnerabilidad de **Confused Deputy**, que permite a un atacante leer o eliminar cualquier archivo en el contenedor de almacenamiento de blobs. Esto se debe a que el servidor confía en la ruta del blob (`blobPath`) proporcionada por el cliente al asociar una imagen con una entidad. Esto puede llevar a la fuga de datos sensibles y a la pérdida de datos para otros usuarios.

Además, se identificaron condiciones de carrera (race conditions) en operaciones de base de datos, lo que podría llevar a estados de datos inconsistentes (p. ej., múltiples imágenes primarias o ninguna). El código también sufre de duplicación significativa entre la lógica de imágenes para `pets` y `cases`, lo que aumenta la carga de mantenimiento y el riesgo de errores.

Se recomienda una refactorización inmediata para eliminar la confianza en el `blobPath` proporcionado por el cliente, implementar transacciones en la base de datos y consolidar la lógica duplicada.

### 2. Riesgos Principales

-   **CRÍTICO: Eliminación Arbitraria de Archivos (Confused Deputy)**
    Un usuario autenticado puede crear una referencia de imagen para una de sus mascotas o casos apuntando a la ruta de un archivo (`blobPath`) perteneciente a cualquier otro usuario. Al eliminar su propia referencia de imagen, el servidor procederá a eliminar el blob en la ruta especificada, causando la pérdida de datos para la víctima.

-   **CRÍTICO: Acceso de Lectura Arbitrario a Archivos (Confused Deputy)**
    De manera similar, un atacante puede asociar el `blobPath` de un archivo privado de otro usuario con una de sus propias entidades. Al solicitar las imágenes de su entidad, el servidor generará una URL de lectura firmada (SAS URL) para el archivo de la víctima, permitiendo al atacante acceder y descargar su contenido.

-   **ALTO: Condiciones de Carrera en la Base de Datos**
    La lógica para actualizar la imagen primaria (`isPrimary`) y para asignar un orden (`sort_order`) no es atómica. Múltiples peticiones concurrentes para la misma mascota o caso pueden resultar en un estado corrupto, como tener varias imágenes marcadas como primarias (o ninguna) o tener valores de `sort_order` duplicados. Esto afecta la integridad de los datos y la experiencia del usuario.

-   **MEDIO: Duplicación de Código Arquitectónico**
    La lógica para gestionar las imágenes de `pets` y `cases` en `server/routes/images.ts` es casi idéntica. Esta duplicación aumenta el esfuerzo de mantenimiento y la probabilidad de que las correcciones de errores o mejoras de seguridad se apliquen en un lugar pero se omitan en el otro.

-   **MEDIO: Modelo de Asociación de Archivos Inseguro**
    El flujo de trabajo actual requiere que el cliente obtenga una URL de subida, suba el archivo y luego informe al servidor qué `blobPath` debe asociar. Este proceso de dos pasos, donde el cliente actúa como intermediario para el `blobPath`, es frágil y la causa raíz de las vulnerabilidades de Confused Deputy.

### 3. Evidencia del Paquete

-   **Confused Deputy (Lectura/Escritura):**
    -   En `server/routes/images.ts`, los endpoints `POST /api/pets/:id/images` y `POST /api/cases/:id/images` aceptan un `blobPath` directamente del cuerpo de la petición (`req.body`).
    ```typescript
    // server/routes/images.ts:121
    const { blobPath, mimeType, originalFilename, sizeBytes, isPrimary } = req.body as { ... };
    // server/routes/images.ts:139
    await query(
      `INSERT INTO pet_images (..., blob_path, ...) VALUES (..., $4, ...)`,
      [..., blobPath, ...],
    );
    ```
    -   El servidor confía ciegamente en este `blobPath` y lo inserta en la base de datos. No hay validación para asegurar que el `blobPath` corresponda al usuario actual o a la entidad que se está modificando.
    -   Posteriormente, `GET /api/pets/:id/images` genera una URL de lectura para cualquier `blob_path` que encuentre en la base de datos para esa mascota.
    ```typescript
    // server/routes/images.ts:181
    url: isStorageConfigured() ? generateReadSasUrl(r.blob_path, 60) : '',
    ```
    -   Asimismo, `DELETE /api/pets/:id/images/:imageId` elimina el blob en la ruta obtenida de la base de datos.
    ```typescript
    // server/routes/images.ts:197
    const { blob_path } = row.rows[0];
    // server/routes/images.ts:200
    await deleteBlob(blob_path).catch(() => {});
    ```
    -   Los tests en `tests/images.test.ts` no cubren este escenario malicioso. Asumen que el `blobPath` proporcionado es legítimo, ocultando el riesgo.

-   **Condiciones de Carrera (Race Condition):**
    -   La actualización de la imagen primaria se realiza en dos consultas separadas sin una transacción.
    ```typescript
    // server/routes/images.ts:133
    if (isPrimary) {
      await query('UPDATE pet_images SET is_primary = FALSE WHERE pet_id = $1', [petId]);
    }
    // ...
    await query('INSERT INTO pet_images ...');
    ```
    -   Si la inserción (`INSERT`) falla después de la actualización (`UPDATE`), la mascota podría quedar sin una imagen primaria.
    -   El cálculo de `sort_order` también es propenso a condiciones de carrera.
    ```typescript
    // server/routes/images.ts:136
    const sortRes = await query('SELECT COALESCE(MAX(sort_order), -1) + 1 AS next FROM pet_images WHERE pet_id = $1', [petId]);
    const sortOrder = sortRes.rows[0].next ?? 0;
    // ...
    await query('INSERT INTO pet_images ... VALUES (..., $9, ...)', [..., sortOrder, ...]);
    ```
    -   Dos peticiones simultáneas podrían obtener el mismo `MAX(sort_order)` y, por lo tanto, intentar insertar con el mismo `sortOrder`.

-   **Duplicación de Código:**
    -   `server/routes/images.ts` contiene bloques de código casi idénticos para `pets` y `cases` para los métodos `POST`, `GET` y `DELETE`. Por ejemplo, `POST /api/pets/:id/images` (líneas 115-161) es funcionalmente equivalente a `POST /api/cases/:id/images` (líneas 211-257).

-   **Manejo de Errores Silencioso:**
    -   La eliminación del blob ignora cualquier error, lo que puede dejar archivos huérfanos en el almacenamiento.
    ```typescript
    // server/routes/images.ts:200
    await deleteBlob(blob_path).catch(() => {});
    ```

### 4. Mitigaciones Sugeridas

1.  **Corregir la Vulnerabilidad de Confused Deputy (Mitigación Principal):**
    -   Modificar el flujo de trabajo de subida de archivos para que el servidor mantenga la autoridad sobre el `blobPath`.
    1.  `POST /api/storage/upload-url` debe generar el `blobPath` en una ubicación temporal (p. ej., `temp/{uid}/{uuid}`). Además de las URLs, debería devolver un **token de subida temporal** (p. ej., un JWT o un UUID seguro almacenado en cache/DB con el `blobPath`).
    2.  `POST /api/pets/:id/images` debe aceptar este `token de subida` en lugar del `blobPath`.
    3.  El servidor debe validar el token, recuperar el `blobPath` temporal asociado y **mover atómicamente el blob** desde la ubicación temporal a su destino permanente (p. ej., `pets/{petId}/{imageId}.jpg`). Esto asegura que la ruta del archivo esté siempre controlada por el servidor y sea correcta para la entidad.

2.  **Utilizar Transacciones en la Base de Datos:**
    -   Envolver las operaciones de base de datos que deben ser atómicas en transacciones (`BEGIN`, `COMMIT`, `ROLLBACK`). Esto aplica a la lógica de `isPrimary` y al cálculo de `sort_order`. Si la función `query` genérica no soporta transacciones, se debe introducir un mecanismo que sí lo haga.

3.  **Refactorizar para Eliminar Duplicación:**
    -   Crear una función o middleware genérico para manejar la lógica de CRUD de imágenes. Esta función podría ser parametrizada con los nombres de las tablas (`pet_images`, `case_images`), nombres de columnas (`pet_id`, `case_id`, `owner_uid`, `created_by`), y el tipo de entidad para centralizar la lógica de autorización, inserción y eliminación.

4.  **Mejorar el Manejo de Errores:**
    -   Reemplazar `.catch(() => {})` con un registro de errores adecuado. Si la eliminación de un blob falla, el sistema debe registrarlo para que los administradores puedan investigar y limpiar los archivos huérfanos.
    ```typescript
    // server/routes/images.ts
    await deleteBlob(blob_path).catch(err => {
      console.error(`Failed to delete blob ${blob_path} for image ${imageId}`, err);
      // Opcionalmente, reportar a un servicio de monitoreo.
    });
    ```

### 5. Victorias Rápidas (Quick Wins)

Aunque se recomienda una refactorización completa, las siguientes acciones pueden mitigar los riesgos más críticos a corto plazo:

1.  **Validar el Prefijo del `blobPath`:** Como medida de contención inmediata, en `POST /api/.../images`, validar que el `blobPath` proporcionado por el cliente comience con un prefijo esperado y seguro, como `temp/{uid}/`, donde `{uid}` es el ID del usuario autenticado. Esto evitaría que un usuario asocie un archivo de la carpeta de otro usuario o de una entidad diferente.
    ```typescript
    // En POST /api/pets/:id/images
    if (!blobPath.startsWith(`temp/${uid}/`)) {
      res.status(400).json({ error: 'validation', message: 'Invalid blobPath' });
      return;
    }
    ```
2.  **Registrar Errores de Eliminación:** Cambiar `await deleteBlob(blob_path).catch(() => {})` a `await deleteBlob(blob_path).catch(console.error)`. Es un cambio de una línea que mejora la observabilidad operativa.
3.  **Validar Formato de `entityId`:** En `POST /api/storage/upload-url`, añadir una validación simple (p. ej., con una expresión regular) para el formato del `entityId` antes de usarlo para construir una ruta.

### 6. Preguntas Abiertas

1.  ¿Cuál es el formato esperado para `entityId` en `pets` y `cases` (p. ej., UUID, CUID)? Conocer el formato permitiría una validación más estricta.
2.  ¿El gestor de base de datos (`db.js`) soporta transacciones? La implementación de las correcciones para las condiciones de carrera depende de la disponibilidad de `BEGIN`/`COMMIT`.
3.  ¿Fue una decisión de diseño intencionada que el cliente gestionara el `blobPath` entre la subida y la asociación? Comprender la intención original podría ayudar a refinar la solución propuesta.
4.  ¿Se pretende que los blobs se compartan entre diferentes entidades (mascotas, casos)? El modelo actual lo permite de forma insegura. Si es un requisito, se necesita un modelo de control de acceso mucho más robusto.
