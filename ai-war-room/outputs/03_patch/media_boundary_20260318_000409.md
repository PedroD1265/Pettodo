### 1. Meta del Parche

El objetivo de este parche es corregir un error crítico de consistencia de datos en la creación de imágenes, mejorar la robustez en la eliminación de blobs y aumentar la cobertura de pruebas para las rutas de la API de imágenes. Los cambios propuestos son mínimos y se centran en la seguridad y la fiabilidad, sin alterar los contratos de la API existentes ni realizar refactorizaciones extensas.

### 2. Cambios Propuestos

Se proponen los siguientes cambios, ordenados por prioridad:

1.  **(Obligatorio) Implementar Transacciones de Base de Datos:** En las rutas `POST /api/pets/:id/images` y `POST /api/cases/:id/images`, las operaciones de `UPDATE` (para desmarcar la imagen primaria anterior) y de `INSERT` (para la nueva imagen) se ejecutan por separado. Si la segunda operación falla, la primera no se revierte, dejando a la entidad sin una imagen primaria. Ambas operaciones deben ser envueltas en una transacción de base de datos para garantizar la atomicidad.

2.  **(Obligatorio) Ampliar Cobertura de Pruebas:** El archivo `tests/images.test.ts` carece de pruebas para la API de imágenes de casos (`Case Images API`), específicamente para el endpoint `DELETE` y para la validación de errores en el endpoint `POST`. Se deben añadir estas pruebas para alcanzar la paridad con las pruebas de imágenes de mascotas y garantizar la fiabilidad.

3.  **(Mejora) Manejo de Errores en la Eliminación de Blobs:** En las rutas `DELETE /.../images/:imageId`, los errores al eliminar el blob de Azure Storage son silenciados con `.catch(() => {})`. Esto puede dejar archivos huérfanos en el almacenamiento si la eliminación falla. Se debe modificar para que, como mínimo, se registre el error en la consola del servidor, facilitando la depuración y el mantenimiento sin interrumpir el flujo del usuario.

### 3. Plan a Nivel de Archivo

El plan se ejecutará en la siguiente secuencia para minimizar riesgos:

1.  **Archivo: `tests/images.test.ts`**
    *   **Acción:** Añadir nuevos casos de prueba para la `Case Images API`.
        *   Un test para `POST /api/cases/:id/images` que envíe un cuerpo de solicitud sin `blobPath` y verifique que se recibe un error de validación `400`.
        *   Un test para `DELETE /api/cases/:id/images/:imageId` que verifique que la imagen se elimina correctamente y se recibe una respuesta `200` con `{ "deleted": true }`.
        *   Un test para `DELETE` que verifique que se devuelve un `404` si se intenta eliminar una imagen que no pertenece al usuario.
    *   **Propósito:** Validar el comportamiento actual y asegurar que las nuevas correcciones se puedan verificar de forma automatizada. Estos tests fallarán inicialmente.

2.  **Archivo: `server/routes/images.ts`**
    *   **Acción 1 (Transacciones):** Modificar los manejadores de `POST /api/pets/:id/images` y `POST /api/cases/:id/images`.
        *   Obtener un cliente del pool de la base de datos (`pool.connect()`).
        *   Envolver las consultas `UPDATE` e `INSERT` dentro de un bloque `try/catch`.
        *   Ejecutar `BEGIN` antes de las consultas, `COMMIT` al final del `try` y `ROLLBACK` en el `catch`.
        *   Asegurarse de liberar el cliente (`client.release()`) en un bloque `finally`.
    *   **Acción 2 (Manejo de Errores):** Modificar los manejadores de `DELETE /api/pets/:id/images/:imageId` y `DELETE /api/cases/:id/images/:imageId`.
        *   Cambiar `await deleteBlob(blob_path).catch(() => {});` por algo como:
          ```typescript
          await deleteBlob(blob_path).catch(err => {
            console.error(`Failed to delete blob ${blob_path}:`, err);
            // No se devuelve un error al cliente para no afectar la experiencia de usuario,
            // pero el error queda registrado para el equipo de desarrollo.
          });
          ```

3.  **Archivo: `tests/images.test.ts`**
    *   **Acción:** Ejecutar de nuevo toda la suite de pruebas.
    *   **Propósito:** Confirmar que los cambios han corregido los problemas, que los nuevos tests ahora pasan y que no se han introducido regresiones en la funcionalidad existente.

### 4. Riesgos y Salvaguardas

*   **Riesgo:** La introducción de la gestión de transacciones manuales puede causar fugas de conexiones si el cliente de la base de datos no se libera correctamente.
    *   **Salvaguarda:** Utilizar un patrón `try/catch/finally` estricto para garantizar que `client.release()` se llame siempre, independientemente de si la operación tuvo éxito o falló. La revisión de código debe verificar explícitamente esta lógica.

*   **Riesgo:** El cambio en el manejo de errores de `deleteBlob` podría, teóricamente, registrar información sensible.
    *   **Salvaguarda:** El cambio propuesto solo registra el `blob_path` y el objeto de error, que no contienen datos del usuario. Es una práctica segura.

*   **Riesgo:** Una implementación incorrecta de la transacción podría bloquear filas en la base de datos o causar fallos inesperados en la API.
    *   **Salvaguarda:** Las pruebas automatizadas, incluyendo las nuevas pruebas añadidas, deben cubrir tanto los casos de éxito como los de fallo para validar que la transacción se comporta como se espera (`COMMIT` en éxito, `ROLLBACK` en error).

### 5. Impacto en las Pruebas

*   **Pruebas Nuevas (Obligatorias):**
    *   Se deben añadir pruebas de integración para la `Case Images API` en `tests/images.test.ts` que cubran:
        1.  `POST` con datos inválidos (respuesta 400).
        2.  `DELETE` de una imagen existente (respuesta 200).
        3.  `DELETE` de una imagen no encontrada o no perteneciente al usuario (respuesta 404).

*   **Pruebas Opcionales (Recomendadas):**
    *   Para una robustez máxima, se podría añadir una prueba que simule un fallo en la consulta `INSERT` (usando `vi.mock`) y verifique que la consulta `UPDATE` anterior se revierte correctamente, confirmando el funcionamiento del `ROLLBACK`.

*   **Pruebas Existentes:**
    *   Todas las pruebas existentes deben pasar sin modificaciones, ya que no se alteran los contratos de la API ni la lógica de negocio principal.

### 6. Recomendación de Despliegue

Los cambios propuestos son internos al backend y no afectan a las interfaces del cliente (`src/app/services/api.ts` permanece intacto). Por lo tanto, el riesgo de impacto para el usuario final es bajo.

*   **Estrategia:** Se recomienda desplegar estos cambios como un único parche.
*   **Rollback (Reversión):** La reversión es sencilla. Consiste en revertir los cambios en el archivo `server/routes/images.ts` a su versión anterior mediante el control de versiones (Git). No se requieren cambios en el esquema de la base de datos ni migraciones, lo que simplifica enormemente el proceso de reversión si fuera necesario. El principal riesgo de no aplicar el parche es la corrupción silenciosa de datos (entidades sin imagen primaria), que este parche precisamente soluciona.
