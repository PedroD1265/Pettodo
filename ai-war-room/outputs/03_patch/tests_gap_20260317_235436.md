# Plan de Parche de PETTODO

## 1. Objetivo del Parche

El objetivo principal de este parche es cerrar la brecha crítica en la cobertura de pruebas para el endpoint de coincidencias de mascotas (`GET /api/matching/cases/:id`). La superficie de código `server/routes/matching.ts` existe y está en uso, pero carece de un archivo de pruebas dedicado, lo que representa un riesgo para la estabilidad y el mantenimiento futuro.

Este plan se centra en añadir pruebas para asegurar el comportamiento del **algoritmo heurístico actual**, garantizando que sea predecible y fiable. Este parche **no** tiene como objetivo implementar el sistema de coincidencias avanzado basado en IA descrito en la documentación canónica (`AI_MATCHING.md`), sino asegurar la implementación existente.

## 2. Cambios Propuestos

La implementación se centrará exclusivamente en añadir código de prueba, sin modificar la lógica de la aplicación existente a menos que se descubra un error crítico durante las pruebas.

### Cambios Requeridos (Must-fix)

1.  **Crear un nuevo archivo de pruebas:**
    *   **Archivo:** `tests/matching.test.ts`
    *   **Propósito:** Contendrá una suite de pruebas completa para el endpoint `GET /api/matching/cases/:id`.

2.  **Desarrollar casos de prueba para cubrir los siguientes escenarios:**
    *   **Autenticación:**
        *   Rechazar solicitudes sin token de autenticación (error 401).
    *   **Validación de Entrada:**
        *   Manejar solicitudes para un `id` de caso que no existe (error 404).
    *   **Lógica de Coincidencias Heurísticas:**
        *   Verificar que se seleccionan los tipos de candidatos correctos (un caso `lost` busca `found` y `sighted`; un caso `found` busca `lost`).
        *   Probar cada componente de la puntuación de forma aislada:
            *   `distanceScore`: Usar datos simulados con coordenadas conocidas para verificar la puntuación por distancia.
            *   `recencyScore`: Usar marcas de tiempo (`created_at`) controladas para validar la puntuación por antigüedad.
            *   `size`: Verificar que la puntuación de tamaño se aplica correctamente cuando los tamaños coinciden.
            *   `colorOverlap`: Probar que la superposición de colores contribuye a la puntuación.
            *   `traits`: Asegurar que los rasgos compartidos aumentan la puntuación.
    *   **Formato de Salida:**
        *   Confirmar que la respuesta contiene los campos esperados (`caseId`, `confidence`, `reasons`, etc.).
        *   Verificar que los resultados se ordenan por `confidence` descendente y se limitan a 20.
    *   **Casos Límite:**
        *   Probar el comportamiento cuando no se encuentran candidatos que cumplan el umbral mínimo de confianza (debería devolver un array vacío).
        *   Probar casos donde los datos de origen o candidato (ej. lat/lng, colores) son nulos.

### Mejoras Opcionales

*   Añadir pruebas unitarias específicas para las funciones auxiliares (`haversineKm`, `recencyScore`, `distanceScore`, `formatTimeAgo`) dentro de `tests/matching.test.ts`. Esto facilitaría la depuración y aislaría aún más la lógica.

## 4. Riesgos y Mitigaciones

| Riesgo                                                                          | Mitigación                                                                                                                                                                                                                                                                                       |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **1. Descubrimiento de bugs en la lógica de coincidencias actual.**             | **Esto es un resultado deseado.** Si las pruebas revelan bugs: <br> 1. Los bugs críticos (ej. crasheos, filtrado incorrecto de tipos de caso) deben ser corregidos como parte de este parche. <br> 2. Bugs menores (ej. inconsistencias leves en la puntuación) deben ser documentados en un ticket para su futura resolución. |
| **2. Las pruebas son incorrectas o incompletas, dando una falsa sensación de seguridad.** | Las pruebas se diseñarán para ser granulares, probando cada componente de la puntuación de forma aislada mediante datos simulados (`mocks`) cuidadosamente diseñados. Se requerirá una revisión de código (peer review) del nuevo archivo `tests/matching.test.ts` por otro desarrollador senior. |
| **3. La creación de datos simulados para la base de datos (`mockQuery`) es compleja.** | Se utilizará un enfoque incremental. Para probar el `distanceScore`, se crearán mocks de dos casos donde solo varíen las coordenadas. Para `recencyScore`, solo variará `created_at`. Este enfoque de "aislamiento de variables" simplifica la creación de mocks y la validación de resultados. |

## 5. Impacto en las Pruebas

Este plan de parche consiste **enteramente** en añadir pruebas. El resultado será un aumento significativo en la cobertura de pruebas y la confianza en una parte crítica del sistema.

*   **Nuevo Artefacto:** Se añadirá `tests/matching.test.ts` al repositorio.
*   **Ejecución:** El script de prueba existente (`"test": "vitest run"`) ejecutará automáticamente los nuevos tests, ya que `vitest` descubre archivos que terminan en `.test.ts` por defecto. No se necesitan cambios en la configuración de CI/CD.

## 6. Recomendación de Despliegue

### Secuencia de Implementación

1.  **Fase 1: Estructura y Autenticación.** Crear el archivo `tests/matching.test.ts` con la configuración inicial (`describe`, `beforeEach`, mocks de `vi`). Implementar las pruebas para los casos de no autenticado (401) y caso no encontrado (404).
2.  **Fase 2: Lógica Central.** Implementar la prueba que verifica la selección correcta de tipos de candidatos (`lost` -> `found`/`sighted`).
3.  **Fase 3: Pruebas de Puntuación.** Añadir pruebas granulares para cada componente de la puntuación (distancia, antigüedad, tamaño, etc.) de forma incremental, una por una.
4.  **Fase 4: Casos Límite y Salida.** Añadir pruebas para el umbral de confianza, la ordenación, el límite de resultados y el manejo de datos nulos.
5.  **Fase 5: Revisión.** Enviar el Pull Request para revisión por pares.
6.  **Fase 6: Merge.** Una vez aprobado, hacer merge a la rama principal.

### Consideraciones de Rollback

Dado que este parche solo introduce archivos de prueba, el riesgo para el entorno de producción es nulo. Un rollback es trivial y consistiría en revertir el commit que añade el archivo `tests/matching.test.ts`. Si se incluye una corrección de un bug, un rollback reintroduciría ese bug, devolviendo el sistema a su estado actual (inseguro pero funcional). El despliegue de este parche es, por lo tanto, de muy bajo riesgo.
