### **Plan de Parcheo PETTODO: `matching`**

### 1. Objetivo del Parche

El objetivo principal de este parche es estabilizar la funcionalidad de `matching` resolviendo inconsistencias críticas y errores entre el backend, el cliente frontend y la lógica duplicada. El sistema actual se basa en heurísticas, no en la IA descrita en la documentación canónica. Este parche se centrará en arreglar el sistema *existente*, no en implementar la visión de IA a futuro.

Los objetivos específicos son:
1.  **Corregir un error de cálculo de confianza** en el endpoint del backend que infla artificialmente las puntuaciones.
2.  **Eliminar la lógica de matching duplicada e inconsistente** del frontend para consolidar la fuente de verdad en el backend.
3.  **Sincronizar el contrato de datos (payload)** entre la respuesta de la API del backend y lo que el cliente frontend espera, para evitar errores en tiempo de ejecución.

### 2. Cambios Propuestos

Se proponen los siguientes cambios, priorizando la estabilidad y la seguridad.

#### Elementos Críticos (Must-fix)

1.  **Backend (`server/routes/matching.ts`):** Corregir el cálculo del `confidence`. El denominador actual (`95`) es incorrecto, ya que la puntuación máxima posible es `115`. Esto causa que los porcentajes de confianza sean más altos de lo que deberían.
2.  **Frontend (`src/app/services/api.ts`):** Actualizar la interfaz `MatchResult` que el cliente espera. Actualmente, hay una discrepancia entre la estructura de datos que devuelve la API y la que el cliente (`matchingApi.run`) promete, lo que probablemente cause fallos en la renderización de resultados.

#### Mejoras y Limpieza (Optional but Recommended)

1.  **Frontend (`src/app/utils/matching.ts`):** Eliminar este archivo por completo. Contiene una reimplementación del algoritmo de matching que es inconsistente con el backend y ya no parece estar en uso por la vía principal (`matchingReal.ts`), generando un riesgo de mantenimiento y confusión.

### 3. Plan a Nivel de Archivo

#### `server/routes/matching.ts`
*   **Acción (Crítica):** Modificar la lógica de cálculo de confianza.
*   **Ubicación:** Línea `149`.
*   **Cambio:**
    *   **Código actual:**
        ```typescript
        const confidence = Math.min(Math.round((score / 95) * 100), 95);
        ```
    *   **Análisis:** La puntuación máxima teórica es `30 (distancia) + 30 (recencia) + 20 (tamaño) + 20 (colores) + 15 (rasgos) = 115`.
    *   **Código propuesto:**
        ```typescript
        const MAX_SCORE = 115;
        const confidence = Math.min(Math.round((score / MAX_SCORE) * 100), 95);
        ```
    *   **Justificación:** Esto normaliza la puntuación de confianza correctamente según la suma de los pesos de las heurísticas, proporcionando una métrica más precisa y honesta.

#### `src/app/utils/matching.ts`
*   **Acción (Mejora):** Eliminar el archivo.
*   **Justificación:** Este archivo contiene lógica de ranking obsoleta y redundante que es ejecutada en el cliente. El adaptador `matchingReal.ts` confirma que la estrategia actual es llamar a la API del backend. Mantener este código es un riesgo.
*   **Acción de seguimiento:** Tras eliminar el archivo, se debe realizar una búsqueda en todo el código base para encontrar cualquier importación de `rankMatches` desde `app/utils/matching` y refactorizar esos componentes para que utilicen el servicio `IMatchingService` (`matchingRealAdapter`).

#### `src/app/services/api.ts`
*   **Acción (Crítica):** Sincronizar la interfaz del resultado del matching.
*   **Ubicación:** La definición de `MatchResult` (probablemente en `src/app/services/interfaces.ts`, pero referenciada aquí).
*   **Problema:** La interfaz `MatchResult` que consume el frontend no coincide con los campos que la API (`/api/matching/cases/:id`) realmente devuelve.
*   **Acción propuesta:** Actualizar (o definir) la interfaz `MatchResult` para que coincida con el payload del backend. Basado en el código de `server/routes/matching.ts`, la interfaz correcta debería ser:
    ```typescript
    // Esta interfaz debe reemplazar la existente para los resultados de matching
    export interface MatchResult {
      caseId: string;
      confidence: number;
      reasons: string[];
      location: string;
      time: string;
      timeLabel: string;
      description: string;
      size: string;
      colors: string[];
      candidateType: 'lost' | 'found' | 'sighted';
      caution: string;
      nextAction: string;
    }
    ```
*   **Justificación:** Este cambio es fundamental para que el frontend pueda procesar y mostrar los resultados de matching sin errores.

#### `docs/02_build/AI_MATCHING.md`
*   **Acción:** No tocar este archivo.
*   **Justificación:** Este documento es canónico y describe la visión del producto a futuro. El parche actual se enfoca en corregir la implementación existente, que es una versión MVP basada en heurísticas. Modificar el documento estaría fuera del alcance y crearía confusión.

### 4. Riesgos y Salvaguardas

*   **Riesgo 1: La corrección de la confianza reduce los porcentajes y filtra más resultados.**
    *   **Descripción:** Al usar el denominador correcto (115 en lugar de 95), todos los puntajes de confianza disminuirán. Esto podría causar que menos candidatos superen el umbral de `confidence < 15` en el backend.
    *   **Salvaguarda:** Este es un efecto esperado y deseado para tener una métrica correcta. Se debe comunicar al equipo de QA y producto. Se debe monitorear después del despliegue si el número de resultados devueltos disminuye drásticamente. El umbral de `15` puede requerir un ajuste si se considera que se están perdiendo demasiados resultados relevantes.

*   **Riesgo 2: Cambios en la interfaz `MatchResult` rompen componentes de la UI.**
    *   **Descripción:** Al actualizar la interfaz `MatchResult`, cualquier componente que la use directamente fallará en la compilación o en tiempo de ejecución.
    *   **Salvaguarda:** La tarea de desarrollo debe incluir la refactorización de todos los componentes de la UI que consumen estos datos para que usen la nueva estructura y sus campos. Se requiere una prueba exhaustiva del flujo de matching de extremo a extremo.

*   **Riesgo 3: La eliminación de `src/app/utils/matching.ts` rompe una funcionalidad desconocida (ej. offline).**
    *   **Descripción:** Es posible, aunque poco probable, que alguna parte de la aplicación utilice esta función para un cálculo síncrono o sin conexión.
    *   **Salvaguarda:** Antes de la eliminación, realizar una búsqueda global en el código fuente para `rankMatches` importado desde `app/utils/matching`. Si se encuentra un uso legítimo, el plan debe ser revisado para alinear la lógica en lugar de eliminarla. Si no se encuentran usos, la eliminación es segura.

### 5. Impacto en las Pruebas

*   **Pruebas Unitarias:**
    *   Se deben actualizar las pruebas unitarias para `server/routes/matching.ts` para que validen el nuevo cálculo de `confidence` con el denominador `115`.
    *   Las pruebas unitarias relacionadas con `src/app/utils/matching.ts` deben ser eliminadas.
*   **Pruebas de Integración:**
    *   Las pruebas de integración para el endpoint `GET /api/matching/cases/:id` deben ser actualizadas para verificar el nuevo schema de respuesta y confirmar que la confianza se calcula correctamente.
*   **Pruebas E2E / Manuales (QA):**
    *   El equipo de QA debe realizar una regresión completa del flujo de matching en la interfaz de usuario.
    *   Verificar que los resultados se muestren correctamente con los datos del nuevo payload.
    *   Prestar especial atención a la percepción del usuario sobre los nuevos puntajes de confianza (más bajos) y si la calidad de los resultados sigue siendo aceptable.

### 6. Recomendación de Despliegue

1.  **Secuencia de Implementación:**
    1.  **Backend (PR 1):** Aplicar el cambio en el cálculo de confianza en `server/routes/matching.ts`. Este cambio puede ser desplegado de manera independiente, ya que solo corrige un valor interno.
    2.  **Frontend (PR 2):** Realizar todos los cambios del frontend en una sola rama/PR:
        *   Eliminar `src/app/utils/matching.ts`.
        *   Actualizar la interfaz `MatchResult` en `src/app/services/api.ts` (o `interfaces.ts`).
        *   Refactorizar todos los componentes de la UI afectados para que se alineen con la nueva interfaz.

2.  **Estrategia de Despliegue:**
    *   Desplegar el PR del backend (1) y el del frontend (2) de manera coordinada y simultánea. Esto asegura que el cliente y el servidor estén siempre sincronizados.
    *   Monitorear los logs del backend para el endpoint `/api/matching/cases/:id` y las métricas de error del frontend inmediatamente después del despliegue.

3.  **Plan de Rollback (Reversión):**
    *   El rollback es sencillo y consiste en revertir los commits correspondientes a los PRs.
    *   Dado que los cambios están acoplados (contrato de datos), si se necesita revertir, se deben revertir tanto el backend como el frontend a sus versiones anteriores para mantener la consistencia.
