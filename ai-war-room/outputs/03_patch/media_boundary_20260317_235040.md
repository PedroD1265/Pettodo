Claro, aquí tienes el plan de parche para la superficie `media_boundary`.

### 1. Objetivo del Parche

El objetivo principal de este parche es mejorar la seguridad, robustez y mantenibilidad del `media_boundary` (gestión de imágenes y almacenamiento). Se enfoca en corregir vulnerabilidades, prevenir condiciones de carrera en la base de datos y aumentar la cobertura de pruebas, todo con cambios mínimos y seguros.

### 2. Cambios Propuestos

Se dividen los cambios en obligatorios (para corregir problemas directos) y recomendados (para mejorar la calidad a futuro).

#### Cambios Obligatorios (Must-Fix)

1.  **Ocultar Mensajes de Error de la Base de Datos:** Actualmente, los endpoints de API exponen mensajes de error de la base de datos al cliente. Esto es un riesgo de seguridad, ya que puede filtrar información sobre la estructura interna del sistema. Se debe modificar para registrar el error real en el servidor y devolver un mensaje genérico al cliente.
2.  **Añadir Pruebas Faltantes:** La API para eliminar imágenes de un `case` (`DELETE /api/cases/:id/images/:imageId`) no tiene cobertura de pruebas. Es necesario añadirla para asegurar su correcto funcionamiento y paridad con la API de `pets`.

#### Mejoras Recomendadas (Opcionales)

1.  **Prevenir Condiciones de Carrera (Race Conditions):** Al crear una referencia de imagen (`POST /.../images`), las operaciones para establecer `is_primary` y calcular `sort_order` se realizan en consultas separadas. Esto podría generar inconsistencias si dos peticiones se procesan simultáneamente. Se recomienda envolver estas operaciones en una transacción de base de datos.
2.  **Registrar Errores de Borrado de Blobs:** Al eliminar una imagen, si falla el borrado del archivo en Azure Blob Storage, el error es ignorado silenciosamente (`.catch(() => {})`). Esto puede dejar archivos huérfanos. Se recomienda registrar este error para facilitar la depuración y mantenimiento.
3.  **Refactorizar Lógica Duplicada:** La lógica para gestionar imágenes de `pets` y `cases` es casi idéntica. A largo plazo, se debería refactorizar en un conjunto de funciones genéricas para reducir la duplicación de código y facilitar el mantenimiento. Este cambio es de mayor alcance y se considera de baja prioridad para este parche.

### 4. Riesgos y Salvaguardas

*   **Cambio de Manejo de Errores:**
    *   **Riesgo:** Bajo. El principal riesgo es que un error legítimo del servidor ya no sea visible en el cliente durante el desarrollo, dificultando la depuración.
    *   **Salvaguarda:** Asegurarse de que el logging del servidor (ej. `console.error`) funcione correctamente y sea monitoreado en los entornos de producción.
*   **Adición de Pruebas:**
    *   **Riesgo:** Nulo. Es una adición que mejora la calidad.
*   **Implementación de Transacciones (Recomendado):**
    *   **Riesgo:** Moderado. Una implementación incorrecta podría causar deadlocks en la base de datos o no liberar conexiones, agotando el pool de conexiones.
    *   **Salvaguarda:** Realizar pruebas exhaustivas, incluyendo pruebas manuales de casos de fallo (p. ej., forzar un error en la consulta `INSERT` y verificar que se ejecuta `ROLLBACK`). Mantener este cambio opcional permite posponerlo si el riesgo se considera alto para un parche inmediato.
*   **Registro de Errores de Borrado de Blob (Recomendado):**
    *   **Riesgo:** Muy bajo. Es solo una línea de log. El único riesgo sería un log excesivamente verboso si los fallos son comunes.
    *   **Salvaguarda:** El logging es la propia salvaguarda contra la pérdida silenciosa de información.

### 5. Impacto en las Pruebas

*   Las pruebas existentes deberían seguir pasando, ya que no verifican el contenido del mensaje de error en caso de un `500 Internal Server Error`.
*   Se deben añadir las nuevas pruebas descritas en la sección `File-Level Plan` para `tests/images.test.ts`.
*   Si se implementan las transacciones, idealmente se deberían añadir pruebas que simulen un fallo a mitad de la transacción para verificar que se realiza el `ROLLBACK`, aunque esto puede ser complejo con el `mocking` actual.

### 6. Recomendación de Despliegue

Se recomienda un despliegue en dos fases para minimizar el riesgo.

1.  **Fase 1 - Inmediata (Cambios Obligatorios):**
    *   Implementar el manejo seguro de errores (ocultar `err.message`).
    *   Añadir la cobertura de pruebas faltante para `DELETE /api/cases/:id/images/:imageId`.
    *   Estos cambios son de bajo riesgo, alta recompensa (cierran una vulnerabilidad) y pueden ser desplegados de forma segura e inmediata.

2.  **Fase 2 - Planificada (Cambios Recomendados):**
    *   Implementar las transacciones en la base de datos y el registro de errores en el borrado de blobs.
    *   Estos cambios requieren más tiempo de desarrollo y pruebas. Deben ser abordados en un sprint posterior como una tarea de mejora técnica.

**Consideraciones de Rollback:**
No hay cambios en el esquema de la base de datos ni en los contratos de la API pública. El rollback para cualquiera de estas fases consiste simplemente en revertir los commits y desplegar la versión anterior del código del servidor. El proceso es de bajo riesgo.
