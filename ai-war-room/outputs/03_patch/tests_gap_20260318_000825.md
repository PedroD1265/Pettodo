# Plan de Parche de PETTODO

## 1. Objetivo del Parche

El objetivo principal es cerrar una brecha crítica en la cobertura de pruebas del backend. Específicamente, la ruta de la API de emparejamiento (`/api/matching/cases/:id`), que es fundamental para la funcionalidad del producto, carece actualmente de pruebas automatizadas. Este parche introducirá un nuevo conjunto de pruebas para validar el comportamiento existente de este endpoint, asegurando su estabilidad y reduciendo el riesgo de regresiones futuras. La implementación se centrará exclusivamente en añadir pruebas, sin modificar el código de producción existente.

## 2. Cambios Propuestos

La propuesta consiste en una única adición al código base, sin modificaciones en los archivos de producción.

- **Creación de un nuevo archivo de pruebas:** Se creará el archivo `tests/matching.test.ts`.
- **Contenido del nuevo archivo:** Este archivo contendrá una suite de pruebas de integración para el endpoint `GET /api/matching/cases/:id`.

Las pruebas cubrirán los siguientes escenarios clave:
- **(Obligatorio) Ruta exitosa:** Una solicitud autenticada con un ID de caso válido que devuelve una lista de candidatos ordenada por relevancia (`confidence`).
- **(Obligatorio) Caso no encontrado:** Una solicitud para un ID de caso que no existe, verificando que la respuesta sea un error `404 Not Found`.
- **(Obligatorio) Fallo de autenticación:** Una solicitud sin un token de autenticación válido, verificando que la respuesta sea un error `401 Unauthorized`.
- **(Recomendado) Sin candidatos:** Una solicitud que encuentra el caso de origen pero no existen casos candidatos compatibles, verificando que la respuesta sea un `200 OK` con un array vacío.
- **(Recomendado) Lógica de puntuación:** Una prueba que verifique que un candidato con alta similitud (ubicación, tamaño, color) recibe una puntuación (`confidence`) más alta que un candidato con baja similitud.

## 4. Riesgos y Salvaguardas

| Riesgo                                        | Descripción                                                                                                                                                                                          | Salvaguarda                                                                                                                                                                                                                                                                                                                                        |
| --------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1. Pruebas que no reflejan la realidad**    | Si las simulaciones (`mocks`) de la base de datos no coinciden con las consultas reales que realiza la ruta, las pruebas podrían pasar pero no garantizar el funcionamiento correcto.                  | La implementación debe seguir de cerca la lógica en `server/routes/matching.ts`. Se debe asegurar que los datos de prueba simulados (casos de origen y candidatos) contengan todos los campos utilizados por el algoritmo de puntuación (`approx_lat`, `approx_lng`, `created_at`, `size`, `colors`, `traits`, etc.) y que las consultas se simulen en el orden correcto. |
| **2. Introducción accidental de cambios**     | Un desarrollador podría verse tentado a "corregir" o "mejorar" la lógica de `matching.ts` mientras escribe las pruebas, lo cual está fuera del alcance de este parche y aumenta el riesgo.            | La revisión de código (Pull Request) debe ser estricta. Solo se aceptarán cambios que añadan el archivo `tests/matching.test.ts`. Cualquier modificación a archivos de producción será rechazada y se deberá mover a una tarea separada. El mantra es: **"Cubrir, no cambiar"**.                                                                             |
| **3. Fragilidad de las pruebas (Test Flakiness)** | Las pruebas que dependen de valores exactos de tiempo (`recencyScore`) o cálculos de punto flotante (`haversineKm`) pueden ser inestables.                                                         | Para la puntuación, en lugar de afirmar un valor de `confidence` exacto (ej. `expect(confidence).toBe(85)`), se debe probar el orden relativo (ej. "el candidato A debe tener una puntuación mayor que el candidato B"). Para el tiempo, se pueden usar valores fijos en los datos de prueba y, si es necesario, simular `Date.now()` para obtener resultados predecibles. |

## 5. Impacto en las Pruebas

El impacto es puramente aditivo y positivo.
- Se añadirá una nueva suite de pruebas (`matching`) al conjunto de pruebas del proyecto.
- La ejecución del comando `npm run test` ahora incluirá la validación del endpoint de emparejamiento.
- La cobertura general de pruebas del backend aumentará, cerrando una brecha de riesgo conocida.
- No se espera que ninguna de las pruebas existentes se vea afectada.

## 6. Recomendación de Despliegue

Se recomienda un despliegue seguro y secuencial.

1.  **Desarrollo:** Implementar el archivo `tests/matching.test.ts` en una rama de git separada (ej. `feature/add-matching-tests`).
2.  **Revisión:** Crear un Pull Request (PR) hacia la rama principal de desarrollo. La revisión debe confirmar que el PR contiene únicamente el nuevo archivo de prueba.
3.  **Fusión:** Una vez aprobado, fusionar el PR. El pipeline de Integración Continua (CI) ejecutará todas las pruebas, incluidas las nuevas, antes de permitir la fusión.
4.  **Despliegue:** Dado que el cambio no afecta al código de producción, no requiere un despliegue de la aplicación en sí. El valor se realiza en el repositorio y en el proceso de CI/CD.

**Consideraciones de Rollback:**
El riesgo de este cambio es extremadamente bajo. Al ser una adición de código de prueba, no afecta la aplicación desplegada. En el improbable caso de que las nuevas pruebas causen inestabilidad en el pipeline de CI (por ejemplo, siendo frágiles), el rollback es tan simple como revertir el commit correspondiente en el control de versiones. Esto no tendría ningún impacto en los usuarios ni en el servicio.
