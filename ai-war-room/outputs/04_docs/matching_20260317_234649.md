Aquí está la auditoría de código versus documentación para la superficie `matching`, en español.

### 1. Resumen de Alineación

La implementación del backend (`server/routes/matching.ts`) se alinea parcialmente con la intención de los documentos, funcionando como un "sistema de sugerencias y revisión" en lugar de un oráculo de identidad. Proporciona con éxito una puntuación, razones, una advertencia de precaución y una siguiente acción, cumpliendo con los requisitos básicos de la salida de cara al usuario.

Sin embargo, existen desviaciones significativas:

1.  **Lógica de Clasificación Duplicada:** Existe una implementación de clasificación casi idéntica pero sutilmente diferente en el frontend (`src/app/utils/matching.ts`). Esto presenta un riesgo importante de inconsistencia y sobrecarga de mantenimiento.
2.  **Pipeline Simplificado:** La implementación actual es un simple proceso de "obtener y puntuar". Carece por completo del pipeline multi-etapa sofisticado descrito en la documentación (por ejemplo, puertas de calidad de imagen, generación de embeddings), que era una promesa clave de la arquitectura.
3.  **Brecha en Detección de Duplicados:** La documentación enfatiza fuertemente el uso del matching para prevenir la creación de registros duplicados. El endpoint actual solo funciona con casos existentes y no hay una implementación visible para una verificación "pre-creación".
4.  **Falta de Pruebas:** La ausencia total de archivos de prueba significa que la corrección de la lógica de puntuación no está verificada y es propensa a regresiones.

En resumen, el backend actual es un punto de partida de MVP funcional que captura el espíritu de la clasificación por sugerencia, pero se desvía en complejidad, omite características documentadas clave y está en riesgo debido a la duplicación de lógica y la falta de pruebas.

### 2. Coincidencias Confirmadas

Las siguientes implementaciones están alineadas con la documentación (`AI_MATCHING.md`):

*   **Sistema de Sugerencia, no de Certeza:** La lógica del backend trata las coincidencias como sugerencias clasificadas con una puntuación de `confidence`, no como una prueba de identidad. Esto se alinea con la regla central de la documentación: "PETTODO matching es un **sistema de sugerencia y revisión**, no un oráculo de identidad automático".
*   **Modelo de Salida Orientado al Usuario:** La respuesta de la API para cada resultado incluye `confidence` (puntuación), `reasons` (razones), `caution` (precaución) y `nextAction` (siguiente acción). Esto cumple directamente con los requisitos de la Sección 7.1 y 9.
*   **Precaución Obligatoria:** Cada resultado de coincidencia incluye una cadena de `caution` codificada, cumpliendo con el requisito de la Sección 9.2: "El marco de precaución es obligatorio".
*   **Clasificación Basada en Múltiples Señales:** El motor de puntuación utiliza una combinación de distancia, antigüedad, tamaño, colores y rasgos, lo cual está alineado con las directrices de la Sección 8.7 para la clasificación de candidatos.
*   **Conciencia del Contexto del Caso:** La lógica determina los tipos de candidatos (`found`, `sighted`) basándose en el tipo del caso de origen (`lost`), lo que demuestra la conciencia del contexto que se pide en la Sección 8.6.

### 3. Desviaciones o Brechas

Aquí se detallan las áreas donde la implementación se desvía de la documentación o donde existen brechas.

*   **Desviación Grave: Lógica de Clasificación Duplicada en el Frontend**
    *   **Observación:** El archivo `src/app/utils/matching.ts` contiene una reimplementación completa de la lógica de clasificación heurística que también existe en el backend. Aunque el adaptador `matchingReal.ts` apunta a la API del backend, la existencia de esta utilidad en el frontend es una fuente importante de riesgo.
    *   **Contradicción:** Las dos implementaciones ya han comenzado a desviarse. Por ejemplo, el backend utiliza `confidence < 15` como umbral de corte, mientras que el frontend usa `confidence >= 20`. Las razones de los rasgos son "matching trait" en el backend y "matching feature" en el frontend. Esto crea dos fuentes de verdad para una funcionalidad crítica.

*   **No Implementado: Pipeline de Coincidencias Multi-etapa**
    *   **Observación:** La documentación (`AI_MATCHING.md`, Sección 8) describe un pipeline detallado con etapas para la puerta de calidad de imagen (Etapa B), la generación de características y embeddings (Etapa E) y la configuración de resultados (Etapa H).
    *   **Brecha:** La implementación actual en `server/routes/matching.ts` omite por completo estas etapas. Realiza una simple consulta a la base de datos para obtener candidatos activos y luego los puntúa directamente. No hay evidencia de puertas de calidad de imagen, normalización o generación de embeddings. La implementación actual es solo la Etapa G (Clasificación).

*   **Desviación: Lenguaje de Cara al Usuario para los Niveles de Coincidencia**
    *   **Observación:** La documentación (Sección 7.2) define explícitamente tres niveles de coincidencia para el usuario: `Probable Match`, `Strong Match` y `No Match`.
    *   **Brecha:** La API del backend devuelve un valor numérico de `confidence` (por ejemplo, de 0 a 95). No realiza la "configuración de resultados de cara al usuario" (Etapa H) para traducir esta puntuación en los niveles aprobados. Esto traslada la responsabilidad (y el riesgo de inconsistencia) al cliente.

*   **Brecha Crítica: Ausencia de Pruebas**
    *   **Observación:** El paquete no contiene archivos de prueba para la lógica de clasificación, ni en el backend ni en el frontend.
    *   **Brecha:** No hay forma de verificar que las funciones de puntuación (`recencyScore`, `distanceScore`, etc.) se comporten como se espera o de protegerlas contra regresiones. Esto oculta desviaciones y hace que el mantenimiento futuro sea arriesgado.

*   **Brecha: Flujo de Detección de Duplicados**
    *   **Observación:** La documentación (Sección 10) dedica una sección completa a usar el matching para la "detección de duplicados" antes de la creación de un nuevo registro.
    *   **Brecha:** El único endpoint de matching proporcionado, `GET /api/matching/cases/:id`, está diseñado para encontrar coincidencias para un caso *ya existente*. No existe una ruta de API o lógica que admita la verificación de posibles duplicados *antes* de que un nuevo perfil o caso sea creado.

*   **Ambigüedad: Puntuación Máxima y Normalización**
    *   **Observación:** La lógica de `confidence` en el backend normaliza la puntuación dividiendo por `95`: `Math.round((score / 95) * 100)`.
    *   **Brecha:** No está claro de dónde proviene el denominador `95` o cuál es la puntuación máxima teórica. Aunque la documentación (Sección 18) permite que las fórmulas exactas no estén finalizadas, este "número mágico" debería ser documentado o derivado de una constante calculada para mejorar la mantenibilidad.

### 4. Riesgos Creados por la Desviación

*   **Experiencia de Usuario Inconsistente:** La lógica de clasificación duplicada significa que un usuario podría ver diferentes puntuaciones de coincidencia o incluso diferentes conjuntos de resultados dependiendo de si el código del frontend o del backend se ejecuta en un flujo de trabajo determinado. Esto erosiona la confianza del usuario.
*   **Deuda Técnica y Sobrecarga de Mantenimiento:** Tener dos fuentes de verdad para la lógica de clasificación es insostenible. Cada corrección de error o mejora debe aplicarse y probarse en dos lugares, lo que duplica el esfuerzo y aumenta la probabilidad de que se separen aún más con el tiempo.
*   **Promesas de Producto Incumplidas:** La documentación vende la visión de un sistema de IA inteligente con puertas de calidad. La implementación actual es mucho más simple y susceptible a resultados de baja calidad si se proporcionan imágenes o datos deficientes ("basura entra, basura sale"). Esto corre el riesgo de hacer que la función parezca "IA falsa" y dañar la reputación de la marca.
*   **Fragmentación de Datos del Sistema:** Al no implementar el flujo de detección de duplicados, el sistema fomentará la creación de registros de animales duplicados, lo que contradice directamente un objetivo principal del sistema de matching. Esto degrada la calidad general de los datos de la plataforma.
*   **Fragilidad y Falta de Fiabilidad:** Sin pruebas, la lógica de puntuación es frágil. Un pequeño cambio en una función de puntuación podría alterar drásticamente las clasificaciones en todo el sistema de manera impredecible.

### 5. Seguimientos Recomendados

1.  **Establecer una Fuente Única de Verdad (Inmediato):** El equipo debe decidir que la clasificación de coincidencias es una responsabilidad exclusiva del backend. El archivo `src/app/utils/matching.ts` debe ser marcado como obsoleto y eliminado lo antes posible para evitar su uso. Todos los flujos de trabajo del frontend deben utilizar `matchingRealAdapter` para llamar al endpoint del backend.
2.  **Agregar Cobertura de Pruebas (Crítico):** Priorizar la escritura de pruebas unitarias para todas las funciones de puntuación heurística (`distanceScore`, `recencyScore`, etc.) en el backend. Además, crear pruebas de integración para el endpoint `GET /matching/cases/:id` para validar la forma general de la respuesta y el orden de clasificación.
3.  **Planificar el Pipeline Documentado:** Reconocer la implementación actual como un MVP de Etapa G. Crear tickets en el backlog para construir las otras etapas documentadas, comenzando con la más impactante: una puerta de calidad de imagen (Etapa B) que pueda filtrar o reducir la puntuación de los casos con imágenes de baja calidad.
4.  **Implementar los Niveles de Coincidencia:** Modificar el endpoint del backend para que, además del puntaje `confidence`, devuelva un campo `matchBand` ("Probable Match", "Strong Match") basado en umbrales documentados. Esto alinea la API con la documentación y reduce la carga de lógica en el cliente.
5.  **Diseñar e Implementar el Endpoint de Detección de Duplicados:** Crear un nuevo endpoint de API (por ejemplo, `POST /api/matching/pre-check`) que acepte los datos de un perfil o caso *no guardado*, ejecute la lógica de clasificación contra los candidatos relevantes y devuelva una lista de posibles duplicados. Esto es esencial para cumplir con un requisito fundamental del producto.
