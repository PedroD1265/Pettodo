### 1. Resumen de Alineación

El sistema de `matching` tiene un endpoint de backend funcional que se alinea con el *espíritu* de la documentación: actúa como un motor de sugerencias, utiliza múltiples señales (ubicación, rasgos, etc.) y devuelve una puntuación con razones y advertencias. Esto cumple con la regla central de ser un "sistema de sugerencia y revisión" en lugar de un oráculo de identidad.

Sin embargo, existe una brecha significativa entre las aspiraciones de la documentación (titulada `AI_MATCHING`) y la implementación actual, que es puramente heurística y no utiliza inteligencia artificial ni análisis de imágenes. Además, existen problemas arquitectónicos notables, como la duplicación de la lógica de puntuación entre el frontend y el backend, y el incumplimiento de varias promesas clave de la documentación, como la detección de duplicados y el uso de bandas de coincidencia de cara al usuario.

### 2. Coincidencias Confirmadas

- **Sistema de Sugerencias:** La implementación se adhiere a la regla principal de la documentación (Sección 2) al funcionar como un sistema de sugerencias y no como una prueba definitiva de identidad. El endpoint devuelve candidatos clasificados para que el usuario los revise.
- **Múltiples Señales:** El algoritmo de puntuación utiliza una combinación de señales como se describe en la documentación (Sección 6), incluyendo la ubicación (`distanceScore`), la antigüedad del reporte (`recencyScore`), el tamaño, los colores y los rasgos.
- **Salida Orientada al Usuario:** El resultado de la API incluye los cuatro componentes obligatorios definidos en la documentación (Sección 7.1 y 9):
    - **Puntuación:** El campo `confidence`.
    - **Razones:** El array `reasons` (p. ej., 'very close to your area', 'similar size').
    - **Advertencia:** Un campo `caution` con un texto fijo.
    - **Siguiente Acción:** Un campo `nextAction` que guía al usuario.
- **Seguridad:** El endpoint está protegido y requiere autenticación (`verifyToken`), asegurando que solo los usuarios registrados puedan ejecutar la búsqueda de coincidencias.
- **Contexto de Casos:** La lógica de negocio distingue correctamente entre casos de "perdido" (`lost`) y casos de "encontrado/avistado" (`found`/`sighted`), buscando solo candidatos del tipo opuesto, lo cual es contextualmente correcto para este flujo.

### 3. Desviaciones o Carencias

- **Ausencia de IA y Análisis de Imágenes:** La desviación más importante. La documentación se titula `AI_MATCHING` y describe un pipeline detallado con puertas de calidad de imagen (Etapa B), extracción de características/vectores de embedding (Etapa E) y similitud de imágenes (Etapa G). La implementación actual (`server/routes/matching.ts`) es puramente heurística y no realiza ningún tipo de análisis de imagen o IA. Compara metadatos de texto y coordenadas, pero ignora por completo el contenido visual de las fotos, una promesa central de la documentación.
- **Lógica de Puntuación Duplicada:** Existe una implementación casi idéntica pero sutilmente diferente de la lógica de puntuación en el frontend (`src/app/utils/matching.ts`) y en el backend (`server/routes/matching.ts`). El adaptador de servicio real (`matchingRealAdapter`) utiliza el backend, lo que convierte al código del frontend en un riesgo de mantenimiento y una fuente de confusión. Por ejemplo, los umbrales de confianza para filtrar resultados son diferentes (frontend `>= 20`, backend `> 15`).
- **Terminología de Coincidencia Faltante:** La documentación exige explícitamente bandas de coincidencia orientadas al usuario como "Probable Match" y "Strong Match" (Sección 7.2). La API solo devuelve un porcentaje numérico de `confidence`. No hay evidencia en el código proporcionado de cómo este número se traduce a las bandas requeridas, ni qué umbrales se utilizan.
- **Promesas de Detección de Duplicados Sin Implementar:** La documentación dedica secciones enteras (10 y 11) a la importancia de la detección de duplicados, especialmente al crear nuevos perfiles de mascotas o "Community Dogs". El endpoint existente solo compara casos de tipos opuestos (`lost` vs `found`) y no parece estar diseñado para el flujo de "crear nuevo perfil y verificar duplicados existentes".
- **Cálculo de `confidence` Opaco:** La puntuación de confianza se calcula con la fórmula `(score / 95) * 100`. El denominador `95` parece ser un número mágico, ya que la puntuación máxima teórica, basada en las funciones del código, es 115 (30 dist. + 30 rec. + 20 tamaño + 20 color + 15 rasgos). Esto hace que la normalización sea confusa e incorrecta.
- **Contrato de API Inconsistente:** La interfaz `MatchResult` definida en el código del frontend (`src/app/utils/matching.ts`) espera un objeto anidado (`case: Case`). Sin embargo, la respuesta real de la API del backend es una estructura plana. Aunque el cliente de la API (`api.ts`) puede manejar esto, indica una falta de sincronización entre las expectativas del frontend y la implementación del backend.

### 4. Riesgos Creados por la Desviación

- **Reclamos de Producto Engañosos:** Comercializar la función como "AI Matching" es una exageración significativa que puede dañar la confianza del usuario. La documentación advierte explícitamente contra parecer un "producto de IA falso con resultados cosméticos", que es precisamente el riesgo que crea la implementación actual.
- **Pesadilla de Mantenimiento:** La lógica de puntuación duplicada significa que cualquier ajuste debe realizarse en dos lugares. Es casi seguro que divergirán con el tiempo, llevando a comportamientos inconsistentes si el código del frontend se utiliza para alguna vista previa o modo sin conexión.
- **Pérdida de Valor del Producto:** La falta de análisis de imágenes significa que el sistema ignora la señal más potente. Un usuario puede subir una foto borrosa o irrelevante y el sistema intentará encontrar coincidencias basándose únicamente en metadatos, perdiendo la promesa clave del producto.
- **Experiencia de Usuario Inconsistente:** Sin una asignación clara del `confidence` a las bandas "Strong/Probable Match", los equipos de UI podrían implementar umbrales arbitrarios, resultando en una experiencia de usuario inconsistente y confusa que no sigue las directrices del producto.
- **Fragilidad del Algoritmo:** El sistema de puntuación heurístico y con números mágicos es frágil. Pequeños cambios en los datos de entrada o la adición de nuevos factores de puntuación pueden desequilibrar todo el sistema sin una forma clara de razonar sobre el impacto.

### 5. Seguimientos Recomendados

1.  **Aclarar la Estrategia de "IA":** El equipo debe decidir si la documentación de `AI_MATCHING` es un documento de visión a futuro o un requisito para el MVP.
    - Si es un requisito del MVP, se debe priorizar la implementación de un análisis de imagen real, aunque sea simple (p. ej., hash perceptual para similitud básica) para cerrar la brecha.
    - Si es una visión a futuro, se debe crear un nuevo documento que describa con precisión el sistema heurístico actual para evitar la confusión y alinear las expectativas.
2.  **Eliminar la Lógica Duplicada:** Auditar el propósito de `src/app/utils/matching.ts`. Si es código muerto, debe ser eliminado. Si tiene un propósito legítimo (p. ej., modo sin conexión), debe ser documentado y su lógica debe ser sincronizada con el backend, que es la fuente canónica de verdad.
3.  **Implementar los Mandatos de la Documentación:**
    - **Bandas de Coincidencia:** Implementar la lógica en el cliente o en un BFF (Backend-For-Frontend) para mapear la puntuación numérica de `confidence` a las bandas "Strong Match" y "Probable Match" requeridas. Documentar los umbrales elegidos.
    - **Detección de Duplicados:** Diseñar e implementar los endpoints y flujos de trabajo necesarios para abordar la detección de duplicados al crear nuevos perfiles de mascotas y "Community Dogs", como se promete en la documentación.
4.  **Refactorizar la Lógica de Puntuación:** Reemplazar el número mágico `95` con una constante calculada o documentada que represente la puntuación máxima esperada. Añadir comentarios en el código (`server/routes/matching.ts`) que expliquen la ponderación y la lógica de normalización.
5.  **Corregir el Contrato de la API:** Asegurarse de que las definiciones de tipos en el frontend que consumen la respuesta de `/api/matching/cases/:id` coincidan con la estructura plana que el servidor realmente devuelve.
