### 1. Resumen de Alineación

En general, la superficie `media_boundary` muestra una buena alineación entre la implementación del backend (`server/routes/images.ts`), la capa de abstracción de almacenamiento (`server/blobStorage.ts`) y los contratos del cliente definidos en `src/app/services/api.ts`. La lógica principal para la emisión de URLs de carga, la asociación de imágenes con mascotas (`pet`) y casos (`case`), y su posterior gestión (listado y borrado) está implementada y es funcional.

Los tests existentes en `tests/images.test.ts` cubren las rutas de éxito y de error más importantes para los endpoints de `pet` y `storage`, lo que confirma el comportamiento básico.

Sin embargo, se han identificado algunas áreas de deriva (drift), ambigüedad y, sobre todo, brechas significativas en la cobertura de tests. Estas brechas ocultan posibles inconsistencias, especialmente en la gestión de imágenes para `case` y en casos de uso más complejos como la gestión de la imagen principal (`isPrimary`) y los fallos en la eliminación de blobs.

### 2. Coincidencias Confirmadas

- **Consistencia de API y Contratos:** Los endpoints definidos en el backend (`/api/storage/upload-url`, `/api/pets/:id/images`, `/api/cases/:id/images`, etc.) coinciden perfectamente con las funciones y tipos definidos en el servicio del frontend (`imageApi` en `api.ts`). Las estructuras de datos como `ImageRef` y `UploadUrlResponse` son consistentes.
- **Verificación de Propiedad:** Todos los endpoints que operan sobre una entidad (`pet` o `case`) implementan correctamente una comprobación para asegurar que el usuario autenticado es el propietario del recurso antes de proceder.
- **Gestión de la Imagen Principal:** La lógica para gestionar la propiedad `isPrimary` está implementada. Al subir una nueva imagen y marcarla como principal, el sistema primero anula esta marca en cualquier otra imagen asociada a la misma entidad, asegurando que solo haya una imagen principal.
- **Ruta de Carga Temporal:** El endpoint `POST /api/storage/upload-url` genera correctamente una ruta de almacenamiento temporal (`temp/<uid>/...`) cuando no se especifica una entidad, permitiendo cargas que se asociarán más tarde.
- **Manejo de Configuración de Storage:** El sistema verifica si el almacenamiento está configurado (`isStorageConfigured`) y devuelve un error `503 Service Unavailable` si no lo está, lo cual es un comportamiento robusto y está cubierto por los tests.
- **Eliminación Completa:** La lógica de `DELETE` elimina tanto el registro de la base de datos (`pet_images` o `case_images`) como el archivo físico (blob) del almacenamiento.

### 3. Deriva o Brechas (Drift Or Gaps)

- **Deriva en el Manejo de Errores de Borrado:**
  - **Implementación:** El código en `DELETE /.../images/:imageId` ignora deliberadamente los errores que puedan ocurrir al eliminar el blob del almacenamiento (`deleteBlob(blob_path).catch(() => {})`).
  - **Problema:** Esto significa que si la eliminación del archivo falla (por permisos, un error transitorio de red, etc.), la API igualmente responderá con éxito (HTTP 200). Esto no está documentado como una decisión de diseño y crea "blobs huérfanos".

- **Duplicación de Código:**
  - **Implementación:** La lógica para los endpoints de imágenes de `pets` es casi idéntica a la de `cases`. Las rutas `POST`, `GET` y `DELETE` están duplicadas, cambiando solo los nombres de las tablas (`pet_images` vs `case_images`) y el campo de propietario (`owner_uid` vs `created_by`).
  - **Problema:** Esta duplicación aumenta el coste de mantenimiento y el riesgo de introducir inconsistencias. Un futuro cambio o corrección de errores podría aplicarse en un lugar y olvidarse en el otro.

- **Brechas en la Cobertura de Tests:**
  - **Faltan Tests para `Case`:**
    - El test para `DELETE /api/cases/:id/images/:imageId` está completamente ausente. Es una ruta crítica que no tiene ninguna cobertura.
    - El test para `POST /api/storage/upload-url` no verifica el flujo cuando se proporciona `entityType: 'case'`.
  - **Tests Incompletos para `isPrimary`:** El test que verifica la lógica de `isPrimary` en `POST /api/pets/:id/images` solo cubre el escenario donde no hay una imagen principal preexistente. No valida que, si ya existe una, esta sea correctamente "degradada".
  - **Tests de Ordenación Ausentes:** Los tests para `GET /.../images` usan un mock que devuelve una sola imagen, por lo que no verifican que la cláusula `ORDER BY sort_order ASC, created_at ASC` funcione como se espera.
  - **Falta Test para Falla de `deleteBlob`:** No hay un test que simule un fallo en `deleteBlob` y confirme que la respuesta de la API sigue siendo `200 OK`. Este test documentaría el comportamiento actual y evidenciaría el riesgo de los blobs huérfanos.

### 4. Riesgos Creados por la Deriva

1.  **Costes de Almacenamiento y Problemas de Datos:** La creación de blobs huérfanos debido a fallos silenciosos en la eliminación incrementará los costes de almacenamiento a lo largo del tiempo. Además, puede generar problemas de integridad de datos y de cumplimiento normativo (p. ej., el "derecho al olvido" del RGPD).
2.  **Inconsistencia de Comportamiento:** La duplicación de código entre la gestión de imágenes de `pets` y `cases` crea un riesgo real de que futuras modificaciones se apliquen solo a una de las entidades, llevando a comportamientos divergentes e inesperados en la API.
3.  **Regresiones Ocultas:** Las brechas en los tests (especialmente la ausencia total del test de eliminación para `cases`) significan que errores o regresiones pueden ser introducidos en el código sin ser detectados por el pipeline de integración continua. Un cambio en la base de datos o en la lógica de negocio podría romper esta funcionalidad sin previo aviso.

### 5. Seguimientos Recomendados

1.  **Clarificar la Política de Eliminación de Blobs:**
    - **Acción:** Discutir con el equipo si el fallo silencioso en la eliminación de blobs es el comportamiento deseado.
    - **Recomendación:** Como mínimo, registrar el error en el lado del servidor para su monitorización, aunque se decida seguir respondiendo con éxito al cliente. Para datos más críticos, considerar una estrategia de reintento o marcar el registro en la base de datos para una limpieza posterior.

2.  **Refactorizar el Código Duplicado:**
    - **Acción:** Abstraer la lógica común para la gestión de imágenes (`POST`, `GET`, `DELETE`) en una función o servicio reutilizable que pueda ser invocado tanto por los controladores de `pets` como de `cases`.
    - **Beneficio:** Esto reducirá la duplicación, simplificará el mantenimiento y garantizará un comportamiento consistente.

3.  **Ampliar la Suite de Tests:**
    - **Acción:** Priorizar la adición de los tests que faltan:
      - Añadir una suite de tests completa para `DELETE /api/cases/:id/images/:imageId`.
      - Añadir un caso de prueba para `entityType: 'case'` en `POST /api/storage/upload-url`.
      - Mejorar los tests de `POST /.../images` para simular una imagen principal preexistente y afirmar que es degradada correctamente.
      - Modificar los tests de `GET /.../images` para que devuelvan múltiples imágenes y verifiquen el orden de la respuesta.
      - Añadir un test que simule un error en `deleteBlob` y afirme que la respuesta sigue siendo `200 OK`, para documentar explícitamente este comportamiento.
