Claro, aquí tienes la auditoría de la superficie `media_boundary`.

### 1. Resumen de Alineación

La implementación de la superficie `media_boundary` está, en general, bien alineada con los contratos definidos en el frontend (`api.ts`). La funcionalidad principal para emitir URLs de carga, persistir referencias de imágenes para mascotas (`pet`) y casos (`case`), y gestionar su ciclo de vida está presente y es funcional.

Sin embargo, se identifica una duplicación de código significativa entre la lógica de imágenes para mascotas y casos, lo que representa un riesgo de mantenimiento. Además, existen brechas en la cobertura de pruebas, especialmente para la entidad `case` y para ciertos comportamientos de respaldo (fallbacks) y manejo de errores, lo que podría ocultar desviaciones o errores.

### 2. Coincidencias Confirmadas

La implementación actual se alinea correctamente con la especificación implícita en los siguientes aspectos:

*   **Contratos de API**: Las rutas, métodos, parámetros y formas de respuesta definidos en `server/routes/images.ts` coinciden con las interfaces y funciones de `imageApi` en `src/app/services/api.ts`.
*   **Seguridad**: Todos los endpoints requieren autenticación y verifican consistentemente la propiedad de la entidad (mascota o caso) antes de realizar cualquier operación.
*   **Generación de URL de Carga**: El endpoint `POST /api/storage/upload-url` funciona como se espera, generando URLs SAS de escritura y lectura. Distingue correctamente entre cargas temporales (almacenadas en `temp/{uid}`) y cargas asociadas a una entidad (ej. `pets/{entityId}`).
*   **Gestión de Imagen Principal**: Al asociar una nueva imagen como principal (`isPrimary: true`), el sistema actualiza correctamente la base de datos para desmarcar cualquier otra imagen que fuera principal para esa misma entidad.
*   **Asociación y Listado**: La creación (`POST`), listado (`GET`) y eliminación (`DELETE`) de referencias de imágenes para mascotas y casos están implementadas y funcionan según lo esperado en los casos de éxito probados.
*   **Borrado de Archivos**: La eliminación de una referencia de imagen en la base de datos (`DELETE .../images/:imageId`) desencadena correctamente el borrado del blob correspondiente en el almacenamiento.

### 3. Desviaciones o Brechas

Se han identificado las siguientes desviaciones, ambigüedades o carencias:

*   **Duplicación de Código**: La lógica de los endpoints para `pet_images` y `case_images` en `server/routes/images.ts` es casi idéntica. Este patrón de "copiar y pegar" no es una desviación de la documentación, pero sí una deficiencia arquitectónica importante que aumenta la complejidad del mantenimiento.
*   **Fallo Silencioso en Borrado de Blob**: El endpoint `DELETE .../:id/images/:imageId` utiliza `await deleteBlob(blob_path).catch(() => {})`. Esto significa que si la eliminación del archivo en Azure Storage falla, la API igualmente devolverá un código `200 OK` al cliente. Esto crea un estado inconsistente donde la referencia en la base de datos se elimina, pero el archivo (blob) queda huérfano en el almacenamiento. Este comportamiento no está documentado ni probado explícitamente.
*   **Brecha en Pruebas para `Case`**: La cobertura de pruebas para los endpoints de imágenes de `case` es significativamente menor que para `pet`.
    *   `DELETE /api/cases/:id/images/:imageId` no tiene ninguna prueba.
    *   `POST /api/storage/upload-url` no prueba la validación de propiedad para `entityType: 'case'`.
*   **Brecha en Pruebas de Valores por Defecto**: La implementación `POST .../images` asigna valores por defecto para `mimeType`, `originalFilename`, `sizeBytes` y `isPrimary` si no se proporcionan. Las pruebas actuales solo cubren el caso en que estos valores se envían explícitamente, dejando sin verificar el comportamiento de fallback.
*   **Brecha en Pruebas de Casos Borde**:
    *   Las funciones auxiliares `safeName` (para sanitizar nombres de archivo) y `blobExt` (para derivar extensiones) no están cubiertas por las pruebas.
    *   No se prueba el comportamiento del sistema cuando las consultas a la base de datos fallan (deberían devolver un error `500`).
    *   La lógica de ordenación (`ORDER BY sort_order ASC, created_at ASC`) en los endpoints `GET` no está verificada en las pruebas.

### 4. Riesgos Creados por la Desviación

Las desviaciones y brechas identificadas generan los siguientes riesgos:

*   **Costos y "Basura" en el Almacenamiento**: El fallo silencioso en el borrado de blobs conducirá a la acumulación de archivos huérfanos en Azure Storage, lo que incrementa los costos y la complejidad de la gestión de datos a largo plazo.
*   **Sobrecarga de Mantenimiento y Errores**: La duplicación de código significa que cualquier corrección de errores o nueva funcionalidad debe implementarse dos veces. Esto aumenta la probabilidad de que las dos implementaciones diverjan con el tiempo, introduciendo errores sutiles en una de ellas.
*   **Errores Ocultos**: Las brechas en la cobertura de pruebas, especialmente para la lógica de `case`, los valores por defecto y la sanitización, implican que podrían existir errores en estas áreas que no serían detectados por el pipeline de CI/CD. Una refactorización futura podría romper fácilmente esta lógica no probada.
*   **Comportamiento Inconsistente**: Si la lógica para `case` y `pet` evoluciona de forma independiente debido a la duplicación, los usuarios podrían experimentar comportamientos diferentes al gestionar imágenes para distintos tipos de entidades, lo que degrada la experiencia de usuario.

### 5. Seguimientos Recomendados

Para mitigar los riesgos y alinear mejor el código con las buenas prácticas, se recomienda:

1.  **Refactorizar la Lógica Duplicada**: Crear una función o servicio genérico que maneje la lógica de negocio para la gestión de referencias de imágenes. Esta función debería ser parametrizable por el tipo de entidad (`'pet' | 'case'`) para eliminar el código duplicado en las rutas.
2.  **Ampliar la Cobertura de Pruebas**:
    *   Añadir pruebas completas para todos los endpoints de `CRUD` de `/api/cases/:id/images`, incluyendo el caso de borrado.
    *   Añadir una prueba para la validación de propiedad de `case` en `POST /api/storage/upload-url`.
    *   Añadir pruebas que omitan campos opcionales en las peticiones `POST` para verificar que los valores por defecto se aplican correctamente.
    *   Añadir pruebas específicas para las funciones `safeName` y `blobExt` con entradas "sucias" (caracteres especiales, sin extensión, etc.).
3.  **Clarificar y Probar el Comportamiento de Borrado**:
    *   Tomar una decisión explícita sobre el comportamiento deseado cuando `deleteBlob` falla. Si el fallo debe propagarse al cliente (con un error `502` o `500`), se debe eliminar el `.catch(() => {})`.
    *   Si el comportamiento actual de "disparar y olvidar" es aceptable, debe documentarse y la prueba correspondiente debe afirmar explícitamente que la solicitud tiene éxito incluso si `mockDeleteBlob` es rechazado.
4.  **Añadir Pruebas de Fallo de Base de Datos**: Simular rechazos en las consultas a la base de datos (`mockQuery.mockRejectedValue(...)`) para asegurar que los endpoints devuelven un estado `500 Internal Server Error` de manera predecible.
