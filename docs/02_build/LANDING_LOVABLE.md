# LANDING_LOVABLE

Purpose:
Define the official landing-page brief for PETTODO so Lovable, ChatGPT, Antigravity, and future collaborators can build and review the landing with consistent product, brand, and messaging alignment.

What belongs here:
- landing goal
- target audiences
- core message
- CTA strategy
- page structure
- allowed and disallowed claims
- visual/content direction
- review criteria for landing iterations

Update rule:
Update when the landing objective, CTA, audience priority, page structure, or core messaging changes.

---

## 1. Landing objective

The PETTODO landing page exists to:

- explain why PETTODO exists
- help users quickly understand the problem PETTODO solves
- show that PETTODO is a real and useful product
- motivate users to try the web app
- communicate value for both pet owners and community-animal care
- create trust through clarity, usefulness, and product seriousness

This landing is not just a promotional page.
It is part of the product entry experience.

---

## 2. Main CTA

### Primary CTA
**Prueba PETTODO ahora**

### CTA destination
The primary CTA should take the user directly to the PETTODO web app.

### CTA placement
The primary CTA should appear:
- in the hero section
- again near the end of the landing

Important:
The landing should be designed so the CTA feels natural and repeated with purpose, not spammy.

---

## 3. Landing status framing

The landing should present PETTODO as a **real web app ready for first users**.

Important:
- do not frame PETTODO as “just an idea”
- do not frame it as “just a pilot”
- do not frame it as an unfinished demo
- do not frame it as a fully mature national-scale platform either

Recommended stance:
PETTODO is a real web app that users can start trying now.

---

## 4. Core landing message

### Main headline
**Una red para cuidar, identificar y recuperar mascotas.**

### Hero subheadline
**Cuida mejor a tu mascota, organiza su información y accede a herramientas inteligentes dentro de una comunidad que protege y cuida a las mascotas.**

### Core messaging principles
The landing must communicate that PETTODO is:
- useful in daily life
- useful in recovery situations
- useful for animal identity and visibility
- useful for pet owners and for controlled community care
- more than a lost-pet app
- more than a pet marketplace
- serious, practical, and community-oriented

---

## 5. Audience priority

### Primary audience
- pet owners

### Secondary audience
- people who help care for dogs in their area / community-animal caregivers

### Tertiary audience
- rescuers

### Fourth-level audience
- shelters

### Fifth-level audience
- allies and veterinarians

Important:
The landing should speak most directly to pet owners first, without erasing the broader community-animal dimension of the product.

---

## 6. What the landing must communicate

The landing must clearly communicate these ideas:

### 6.1 Everyday pet value
PETTODO is useful before emergencies.
It helps organize pet information and improve daily care.

### 6.2 Digital identity
PETTODO helps build a useful digital identity for each animal through:
- profile
- information
- history
- QR
- visibility

### 6.3 Recovery usefulness
If a pet is lost, PETTODO helps act faster through:
- cases
- QR
- public visibility
- intelligent tools
- community participation

### 6.4 Community dimension
PETTODO supports not only owned pets, but also controlled visibility and care for community dogs / street-animal situations.

### 6.5 Smart tools
PETTODO includes useful intelligent tools as part of the experience, but should not be framed as “AI hype.”

### 6.6 Safety and trust
PETTODO protects sensitive information and uses safer contact logic instead of careless public exposure.

---

## 7. What the landing must NOT communicate

The landing must not suggest that PETTODO is:

- only a lost-pet app
- only a pet-management tracker
- a finished nationwide platform already deployed at scale
- a mobile-native app already fully released
- an AI-first novelty product
- a full services marketplace already mature
- an unrestricted social network
- an uncontrolled public database of street dogs

The landing must also avoid implying:
- that AI confirms identity with absolute certainty
- that all community dogs are the same as all street dogs
- that owner information is publicly exposed by default

---

## 8. Content structure

The landing should use a scrolling structure with clear visual hierarchy.

## 8.1 Hero
Goal:
Create interest quickly and drive users into the app.

Must include:
- main headline
- subheadline
- primary CTA
- visual proof of product
- trust-building first impression

## 8.2 Problem / context section
Goal:
Sensitize the user and explain why PETTODO matters.

This section should include:
- relevant context data
- clear explanation of the problem
- emotional but structured framing
- the difference between pet ownership, loss risk, and community-animal realities

Important:
This section should inform and move the user, not manipulate them.

## 8.3 Core product features section
Goal:
Show the most interesting and useful functions.

This section should include examples like:
- organized pet information
- digital identity
- QR
- intelligent tools
- matching support
- automatic flyer/post preparation
- training/helpful features
- practical utility in daily use

This section should focus on strong, high-value features.

## 8.4 Community dogs / street-animal visibility section
Goal:
Show that PETTODO also helps give visibility, care, and better control to dogs in public/community spaces.

Important:
This is a strong section, but not the main identity of the hero.

This section should communicate:
- visibility
- responsible support
- evidence-based care actions
- community participation
- controlled, non-chaotic handling of shared animal records

## 8.5 More product functions / ecosystem section
Goal:
Show that PETTODO can grow into a broader product without overexplaining every feature.

This section should briefly reference:
- additional useful functions
- training
- services
- community-related tools
- future ecosystem breadth

Important:
Keep this section brief and clear.
Do not let it dilute the core product.

## 8.6 Safety / trust section
Goal:
Explain why PETTODO is more trustworthy than fragmented social posting.

This section should highlight:
- protected contact
- organized pet information
- safer visibility
- useful digital records
- controlled handling of community-animal data

## 8.7 Allies / support section
Goal:
Invite support and contact from organizations, collaborators, and people who want to help or need help.

This section should communicate something like:
- “¿Quieres apoyarnos o necesitas apoyo?”
- PETTODO is open to useful collaboration

It should feel short, direct, and clear.

This section must include a form with:
- name
- contact number
- Gmail/email
- comment/message
- submit button

## 8.8 Final CTA section
Goal:
Give the user one more clear chance to try the app.

Should repeat the main CTA:
**Prueba PETTODO ahora**

## 9. Community Dogs positioning in landing

Community Dogs / street-animal visibility is important, but should not dominate the first impression of the product.

Landing rule:
- do not make Community Dogs the hero
- do give them a strong dedicated section
- do frame them as a meaningful and responsible part of PETTODO
- do not let PETTODO look like it exists only for stray/street-animal management

---

## 10. AI positioning in landing

AI should appear as a useful and differentiated capability, but not as the entire identity of the product.

### AI role in landing
The landing may communicate that PETTODO uses intelligent tools for things like:
- recognition support
- matching support
- preparation of posts/flyers
- smart assistance

### AI framing rule
Do not present PETTODO as an “AI gimmick.”
Do not make AI the hero of the brand.
Do not imply that AI replaces all human confirmation or community logic.

Recommended positioning:
AI is an important support layer inside a practical and trustworthy product.

---

## 11. Trust and proof strategy

The first landing version will not rely on large partner logos or fake credibility markers.

Instead, trust should be built through:
- clarity of the problem
- seriousness of the product
- useful screenshots/mockups
- privacy-aware messaging
- protected contact logic
- structured explanation of value
- strong brand coherence
- credible product tone

Later versions may include:
- allies
- shelters
- veterinarians
- institutions
- metrics
- testimonials

But these should not be invented or forced before they exist.

---

## 12. Data / statistics section

The landing should include a contextual data section that:
- informs
- explains the scale of the problem
- adds emotional weight
- supports the need for PETTODO

### Presentation style
Use:
- context blocks
- counters
- organized data cards
- visually clear comparisons

### Possible examples
- total number of dogs
- total number of dogs in the city
- size of the difference / gap
- community/street-animal context
- pet-loss or related context

Important:
Only use verified and documented numbers.
Do not invent statistics.
Do not use random emotionally manipulative claims.

---

## 13. Visual direction for landing

The landing should use a balance of:

- product mockups / UI previews
- screenshots or idealized product views
- emotionally relevant pet photography
- clean modern layout

### Visual rule
The landing should feel:
- real
- modern
- useful
- sensitive without being melodramatic
- community-aware
- trustworthy

### Avoid
- over-illustration
- childish pet aesthetics
- empty “startup gradients everywhere”
- random stock icon overload
- overly emotional rescue-poster style

## 17. Claims policy

### Allowed claim directions
The landing may claim that PETTODO helps users:
- care for pets better
- organize pet information
- build digital identity for animals
- access QR/public-profile logic
- use intelligent tools
- act better in recovery situations
- give responsible visibility to community dogs

### Not allowed
Do not claim:
- guaranteed pet recovery
- perfect AI recognition
- nationwide full deployment
- full marketplace maturity
- universal partner support already active
- complete public-animal control system already proven at scale

---

## 18. Tone and copy direction

The landing must follow the brand tone:

### More warm / engaging in:
- everyday pet care
- empathy
- user motivation
- community language

### More structured / trustworthy in:
- safety
- sensitive features
- community dogs
- visibility rules
- contact logic
- smart tools
- data framing

The copy should avoid:
- sounding childish
- sounding like a rescue NGO flyer
- sounding too technical
- sounding like exaggerated startup hype

---

## 19. Review criteria for Lovable iterations

Any landing iteration must be reviewed against these questions:

### Message
- Does it correctly describe PETTODO?
- Does it avoid making PETTODO look like only a lost-pet app?
- Does it avoid overselling AI?
- Does it avoid promising functions that are not ready?

### Brand
- Does it follow `BRAND_SYSTEM.md`?
- Does it use the logo and colors correctly?
- Does it feel trustworthy + human?

### Product alignment
- Does it match `MASTER_CONTEXT.md` and `CURRENT_STATE.md`?
- Does it reflect the real MVP direction?

### Structure
- Is the CTA clear?
- Is the problem explained?
- Are the strongest features visible?
- Is Community Dogs included in the right level?
- Is the ally/support section clear?

### Visual quality
- Is the layout clean and modern?
- Is the hierarchy clear?
- Does it balance UI/product visuals and emotional imagery well?
