**Date:** 2026-03-18

**Status:** Key removed from current workflow. Forensic verdict: **DUMMY_FIXTURE**

---

## Verdict

**DUMMY_FIXTURE**

**Confidence level:** High

---

## Executive Summary

The RSA private key committed to `.github/workflows/ci.yml` in commit `9578177` was a dummy CI fixture — a locally generated key inserted intentionally to satisfy the workflow's `FIREBASE_PRIVATE_KEY` environment variable without using any real Firebase credential. It was not connected to any real Firebase project or service account.

The key has been removed from the current workflow (replaced with `${{ secrets.FIREBASE_PRIVATE_KEY }}`). No real credentials need to be rotated as a result of this exposure. The key could not have authenticated against any real Firebase project.

### E. Critical format mismatch: wrong key type for Firebase

**This is the strongest technical evidence.**

Google Cloud / Firebase service account private keys are **always** in PKCS#8 format:

```
-----BEGIN PRIVATE KEY-----
```

The key committed to `ci.yml` is in **PKCS#1** format:

```
-----BEGIN RSA PRIVATE KEY-----
```

- **Real Firebase service account key format:** `BEGIN PRIVATE KEY` (PKCS#8, per Google's key issuance system)
- **Committed key format:** `BEGIN RSA PRIVATE KEY` (PKCS#1, bare RSA key, not generated by GCP)

The `.env.example` file in this same repo correctly documents the expected format as `BEGIN PRIVATE KEY` (PKCS#8), confirming awareness of the correct format:

```
# FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
```

A real Firebase service account key would never bear the `BEGIN RSA PRIVATE KEY` header. GCP's key generation system exclusively emits PKCS#8-wrapped keys. The wrong header is definitive proof this key did not come from a Firebase service account.

### F. Cryptographic analysis of the key fragment

The committed fragment:
```
MIIEpAIBAAKCAQEA0Z3VS5JJcds3xHn/ygWep4J0CYXL8G1dQHZdGnF7kTZdlMhW
```

DER structure analysis:
- `0x30 0x82` → Valid DER SEQUENCE header
- `0x04 0xa4` → DER content length = 1188 bytes (total key: 1192 bytes)
- `0x02 0x01 0x00` → RSAPrivateKey version = 0 (valid PKCS#1 structure)
- `0x02 0x82 0x01 0x01` → Modulus INTEGER, length = 257 bytes = **2048-bit RSA**

The key fragment is cryptographically well-formed: it is a real, properly generated 2048-bit RSA private key in PKCS#1 format. It was not randomly invented or corrupted. However, it is **not a Firebase credential** — it is a locally generated RSA key used as a stand-in fixture to satisfy the env var without providing a real service account credential.

## Technical Reasoning: Why These Are Not Real Credentials

1. **Wrong key format:** Firebase/GCP issues PKCS#8 keys. This is PKCS#1. Incompatible.
2. **Wrong project ID:** `ci-project` is not a valid or existing GCP project.
3. **Wrong email format:** `ci@ci.iam.gserviceaccount.com` — the domain segment `ci` is not a real GCP project. Real accounts end in `@<project-id>.iam.gserviceaccount.com`.
4. **Localhost database:** `postgresql://ci:ci@localhost:5432/ci` — confirms this was never intended for a real environment.
5. **Engineering instruction:** The prompt that generated this workflow explicitly required "dummy env vars."
6. **Project documentation:** The team's own QA record calls these "dummy env vars."
7. **Tests fully mock Firebase:** The key value is never cryptographically evaluated in CI.
