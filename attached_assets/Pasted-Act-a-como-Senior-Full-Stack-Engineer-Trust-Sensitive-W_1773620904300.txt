## Implementa en este bloque

### A. Extensión de schema / persistencia
Agrega tablas nuevas mínimas:
- protected_contact_threads
- protected_contact_messages
- protected_contact_events
- community_dogs
- community_dog_sightings
- community_dog_actions
- change_requests
- evidence_items
- review_decisions
- abuse_flags
- audit_logs

### B. Estados mínimos
Congela y usa estos estados:
- review_state: pending_review | approved | rejected
- verification_state: pending_review | approved | rejected
- thread_status: open | closed | blocked
- reveal_state: not_revealed | revealed | revoked

### C. Relaciones mínimas
- protected_contact_threads debe poder vincularse a pet_id y opcionalmente case_id
- community_dog_sightings y community_dog_actions deben vincularse a community_dog_id
- evidence_items debe usar target_entity_type + target_entity_id
- change_requests debe usar target_entity_type + target_entity_id
- review_decisions debe registrar reviewer + decision + notes
- audit_logs debe poder registrar acciones sensibles

### D. Rutas / API mínimas
Implementa endpoints mínimos para:
- POST /api/protected-contact/threads
- GET /api/protected-contact/threads/:id
- POST /api/protected-contact/threads/:id/messages
- POST /api/protected-contact/threads/:id/reveal-request
- POST /api/protected-contact/threads/:id/reveal-decision

- POST /api/community-dogs
- GET /api/community-dogs/:id
- POST /api/community-dogs/:id/sightings
- POST /api/community-dogs/:id/actions

- POST /api/change-requests
- POST /api/evidence-items

- GET /api/reviews/pending
- POST /api/reviews/:entityType/:entityId/approve
- POST /api/reviews/:entityType/:entityId/reject

- POST /api/abuse-flags

### E. Reglas de acceso mínimas
- crear/contactar/contribuir/revisar requiere auth
- owner-only para reveal decision cuando aplique
- moderator/operator-only para approve/reject review routes
- no high-impact anonymous actions
- usa el auth system actual; no crees auth separado para moderación

### F. Reglas de salida pública/privada mínimas
Asegura desde backend:
- nunca exponer phone/email/address/exact coordinates en rutas públicas
- QR/public pet flow debe devolver protected contact entry point, no contacto directo por defecto
- Community Dogs solo deben ser públicos si review_state = approved
- pending/rejected/review metadata/audit metadata/abuse flags deben permanecer internos
- usa approximate/public-safe shaping, no raw internal fields

### G. Auditabilidad mínima
Registra al menos:
- contact initiated
- message sent
- reveal requested
- reveal granted/revoked
- community dog creation submitted
- sensitive change requested
- evidence submitted
- review approved/rejected
- abuse flag created
