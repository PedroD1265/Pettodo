## Problema crítico actual
Hoy, crear una mascota desde la UI en integration mode puede mostrar éxito en UI/local state, pero no persistir realmente en PostgreSQL.

El runtime report atribuye la causa más probable a este patrón:
- AppContext.addPet() hace optimistic update
- luego dispara petApi.create() async
- el fallo puede quedar silencioso
- la mascota parece existir en sesión actual
- al refrescar o volver a entrar desaparece porque la DB nunca la tuvo

Además, el mismo riesgo probablemente aplica también a updatePet y deletePet.

## Objetivo exacto
Hacer que pet CRUD en integration mode sea confiable y server-authoritative:

- create/update/delete deben depender del resultado real del backend
- no debe existir éxito silencioso ni éxito falso
- si la API falla, el usuario debe verlo claramente
- refresh y nueva sesión deben reflejar exactamente la DB
- la respuesta del servidor debe ser la fuente autoritativa del estado final en UI
- demo mode debe seguir funcionando, separado y explícito

## Desired implementation direction
Preferencia principal:

En integration mode:
- await create/update/delete API calls
- solo después del success actualizar UI final
- usar el registro devuelto por el servidor como autoridad
- si la API falla, mostrar error visible y mantener estado veraz
- si hace falta, refetch desde API después de mutación para asegurar consistencia

Puedes mantener optimistic UI solo si queda explícitamente “pending” y luego se confirma o revierte visiblemente.
Pero para este packet, la preferencia es evitar optimistic success en integration mode.
